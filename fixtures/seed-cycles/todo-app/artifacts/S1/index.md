## Brief (Phase A — 確定済)

### 目的 (1文)
個人が日常のタスクを素早く登録・整理して「今日やること」を手放せる、1人用のシンプルな ToDo 管理アプリを作る。

### 対象ユーザー
- **主要ペルソナ**: 自分のタスクをメモやスプレッドシートで管理しているが、優先順位の整理や完了の可視化が煩雑に感じている個人ユーザー(エンジニア・デザイナー・学生など)
- **利用文脈**: PC ブラウザで作業しながら、その日のタスクを随時追加・完了チェックしていく
- **困りごと**: タスクが増えると「どれが終わっていないか」「期限が近いものはどれか」が一目でわからない

### スコープ
#### やる
- タスクの追加(タイトル必須、期限日任意)
- タスクの完了チェック(完了状態のトグル)
- タスクの削除
- 未完了 / 完了 のフィルタリング
- 期限順 / 作成順 の並び替え

#### やらない(明示)
- チーム共有 / 複数ユーザー対応
- プッシュ通知 / リマインダー
- 繰り返しタスク
- サブタスク / タスク階層化
- タグ / ラベル管理
- データのエクスポート

### 最初のリリースで外せない1機能
タスク一覧画面で未完了タスクを一覧表示し、完了チェックできること。

### 質疑応答ログ

#### Q-01 — データはどこに保存するか(ブラウザローカル vs サーバー)
- **回答**(人間の回答を AI が記入):
  > ブラウザの localStorage に保存する。サーバー不要、ログイン不要でよい。
- **確定**(AI 記入):
  > localStorage を唯一のデータストアとして使用する。ページをリロードしてもデータが消えないことを保証する。

#### Q-02 — 期限日が過ぎたタスクの扱い
- **回答**(人間の回答を AI が記入):
  > 特別な扱い(エラー表示など)は不要。普通に一覧に表示されればよい。
- **確定**(AI 記入):
  > 期限超過タスクに対して警告表示・自動削除・スタイル変化は行わない。表示は通常と同じ。

#### Q-03 — タスクの編集(タイトル・期限の変更)機能は必要か
- **回答**(人間の回答を AI が記入):
  > 今回はなくてもよい。削除して再追加でよい。
- **確定**(AI 記入):
  > タスク編集(インプレース編集)は v0.0.1 スコープ外とする。US に含めない。

### AI が独自に決めたこと と 理由

#### D-01 — US 粒度の基準を「画面操作1往復で価値が完結する単位」に設定
- **理由**: S1 スキルの粒度定義「1 US = 1つの独立してテスト可能な縦スライス」に従い、タスク管理の各操作(追加・完了・削除・フィルタ・並び替え・期限設定・永続化)を独立した US とした。まとめて1 US にすると受け入れ条件が曖昧になる。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

#### D-02 — サーバーレス(localStorage)構成を前提として US を書く
- **理由**: Q-01 で確定。データ永続化の US は localStorage の読み書き前提で AC を書く。
- **種別**: 事業判断(Q-01 で human-gate 済)
- **上書き**: なし

## US-01: タスクの追加

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
タスクを登録する手段がないとアプリとして成立しない。ユーザーが「今日やること」を思いついた瞬間に素早く入力できることが、ツールを使い続けてもらう最初のハードルを下げる。

#### UX へのインパクト
テキスト入力欄とボタン1つで完結し、入力後すぐに一覧に追加される。余計な遷移がなく、思考を中断せずタスクを記録できる。

#### 受け入れ条件 (AC)
- タスクタイトルを入力して「追加」ボタン(またはEnterキー)を押すと、タスクが一覧の末尾に追加される
- タイトルが空欄の場合は追加されず、エラーメッセージが表示される
- 追加後、入力欄は空文字にリセットされる
- 追加されたタスクは初期状態として「未完了」になる

### AI が独自に決めたこと と 理由

#### D-01 — 期限日の設定は別 US(US-05)に分離
- **理由**: タスク追加と期限設定は独立してテスト可能。追加時に期限を省略できる仕様のため、別スライスとして切り出すのが自然。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## US-02: タスクの完了チェック

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
「やること」を「やった」に移す操作がアプリの中心的な価値。完了チェックがないと進捗が見えず、タスク管理ツールとして機能しない。

#### UX へのインパクト
チェックボックス1クリックで完了状態をトグルできる。完了済みタスクは視覚的に区別され(打ち消し線など)、達成感が得られる。未完了に戻すことも可能。

#### 受け入れ条件 (AC)
- 各タスク行にチェックボックスが表示される
- チェックをオンにするとタスクの状態が「完了」に変わり、タイトルに打ち消し線が付く
- 完了状態のチェックをオフにすると「未完了」に戻り、打ち消し線が消える
- 完了状態はページリロード後も保持される(localStorage 経由)

## US-03: タスクの削除

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
不要なタスクを一覧から消せなければリストが増え続ける。ユーザーが自分の一覧を最新の状態に保てることが、ツールを使い続ける前提条件。

#### UX へのインパクト
削除ボタン1つで即座にタスクが消える。削除は取り消しできないため、誤操作リスクを考慮してボタンはタスク行の右端に配置し、視認性を控えめにする。

#### 受け入れ条件 (AC)
- 各タスク行に削除ボタン(「削除」または ✕ アイコン)が表示される
- 削除ボタンを押すと該当タスクが一覧から即座に削除される
- 削除されたタスクは localStorage からも削除され、リロード後も復元されない
- 削除後、他のタスクの順序と状態は変わらない

### AI が独自に決めたこと と 理由

#### D-01 — 削除確認ダイアログは設けない
- **理由**: 「シンプルさ」がプロダクトの中心価値。1人利用で誤削除リスクは低く、確認ダイアログは操作の流れを断ち切る。v0.0.1 スコープ外。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## US-04: 未完了 / 完了のフィルタリング

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
タスクが増えると完了済みのノイズが邪魔になる。「今やるべきこと」だけを見たいユーザーと「何を終わらせたか」を振り返りたいユーザーの両方に対応するため、表示を絞り込む手段が必要。

#### UX へのインパクト
「すべて」「未完了」「完了」の3択をタブまたはボタンで切り替えられる。切り替え後も現在の選択状態が視覚的に示される。

#### 受け入れ条件 (AC)
- 画面上部に「すべて」「未完了」「完了」の3つのフィルタボタンが表示される
- 「未完了」を選択すると完了チェック済みタスクが非表示になる
- 「完了」を選択すると未完了タスクが非表示になる
- 「すべて」を選択すると全タスクが表示される
- 現在選択中のフィルタが強調表示される
- フィルタ切り替えはタスクのデータを変更しない(表示の絞り込みのみ)

## US-05: 期限日の設定

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
「いつまでに」という情報がないと優先順位がつけにくい。期限日を任意で付加することで、ユーザーは締め切りを意識しながらタスクを管理できる。

#### UX へのインパクト
タスク追加フォームに日付入力欄を追加する。日付は入力しなくても追加できる(任意)。設定した期限日はタスク行に表示され、一目で確認できる。

#### 受け入れ条件 (AC)
- タスク追加フォームに日付入力欄(date ピッカー)が表示される
- 日付は任意項目で、空のまま追加しても問題ない
- 期限日を設定した場合、タスク一覧行に「期限: YYYY/MM/DD」形式で表示される
- 期限日を設定しなかった場合、期限の表示エリアは空白になる
- 設定した期限日は localStorage に保存され、リロード後も保持される

### AI が独自に決めたこと と 理由

#### D-01 — 期限日の表示形式を「YYYY/MM/DD」に固定
- **理由**: ローカライズやユーザー設定は v0.0.1 スコープ外。日本語ユーザーが直感的に読める形式として YYYY/MM/DD を採用。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## US-06: 並び替え

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
タスクが増えると「どれが一番急ぎか」が埋もれる。期限順に並べることで緊急タスクをすぐに発見でき、作成順に戻すことでタスクの追加履歴を把握できる。

#### UX へのインパクト
並び替えドロップダウンで「作成順(デフォルト)」「期限が近い順」を選択できる。切り替えはリアルタイムで反映される。

#### 受け入れ条件 (AC)
- 画面上部に並び替え選択UI(ドロップダウンまたはボタン)が表示される
- 「作成順」を選ぶと追加した順(古い順)でタスクが並ぶ
- 「期限が近い順」を選ぶと期限日が早いタスクが上に来る
- 期限日が未設定のタスクは「期限が近い順」の場合、期限あり全タスクの後ろに表示される
- 並び替えはタスクのデータを変更しない(表示順の変更のみ)

## US-07: データの永続化

### メタ
- ステータス: 確定

### 3 視点

#### なぜするか (Why)
ページをリロード・閉じて再度開いたときにタスクが消えると、ツールとして信頼されない。データを永続化することで、ユーザーは安心して登録・管理できる。

#### UX へのインパクト
ユーザーが意識しなくてもタスクが保持される。特別な「保存」操作は不要で、すべての変更(追加・完了・削除・期限設定)は即時自動保存される。

#### 受け入れ条件 (AC)
- タスクの追加・完了状態の変更・削除・期限日の変更はすべて localStorage に即時反映される
- ページをリロードすると直前に表示していたタスク一覧(内容・状態・期限日)が復元される
- ブラウザを閉じて再度開いてもタスクが復元される
- localStorage のキーは `todo-app-tasks` とし、JSON 配列として保存される
- タスク1件のデータ構造は `{ id: string, title: string, done: boolean, dueDate: string | null, createdAt: string }` とする

### AI が独自に決めたこと と 理由

#### D-01 — localStorage キーを `todo-app-tasks` に固定
- **理由**: アプリが単一で他アプリとのキー衝突リスクが低い。プレフィックス付き命名で将来的な衝突も最小化。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

#### D-02 — タスク1件の id は UUID(crypto.randomUUID)で生成
- **理由**: 連番 ID は削除時に管理が煩雑。ブラウザネイティブの `crypto.randomUUID()` で依存ゼロの一意 ID を生成する。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## 棄却した US 案

### R-01 — タスク編集(タイトル・期限の変更)
- **棄却理由**: Q-03 で「削除して再追加でよい」と確認。v0.0.1 のシンプルさを優先し不採用。

### R-02 — 完了タスクの一括削除
- **棄却理由**: US-03 の個別削除で代替可能。一括削除は誤操作リスクが高く、v0.0.1 の価値提供に不要と判断。

### R-03 — 期限超過タスクの強調表示(赤字・警告)
- **棄却理由**: Q-02(期限切れ扱い)で「特別扱い不要」と確認。スタイル差分がないことで実装コストを下げる。
