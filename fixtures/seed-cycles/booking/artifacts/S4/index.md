## アーキテクチャ方針

### 全体構成
- **構成方式**: SPA + REST API(Node.js/Express)
- **レイヤー分離**:
  - フロントエンド: UI コンポーネント / ページ / API クライアント(React Query)
  - バックエンド: ルーター(Express routes) / サービス層(ビジネスロジック + ダブルブッキングチェック) / リポジトリ層(Prisma クライアント)
  - DB: PostgreSQL(予約テーブル・会議室テーブル)

### 状態管理
- **サーバー状態**: React Query — 予約一覧・会議室一覧・カレンダーデータのキャッシュと再検証
- **クライアント状態**: React useState / useReducer — フォーム入力・モーダル開閉など局所的な UI 状態のみ
- **URL 状態**: 会議室 ID と表示日付をクエリパラメータで管理(`?roomId=1&date=2026-06-10`)。ブックマークとブラウザバックに対応

### エラーハンドリング
- **リトライ方針**: React Query の自動リトライはデフォルト 3 回。ダブルブッキングエラーは 409 Conflict で返し、クライアントはリトライせずユーザーに再入力を促す
- **フォールバック方針**: API タイムアウト時はスケルトンをそのまま維持し「読み込みに失敗しました」バナーを表示
- **エラー表示方針**: フォームバリデーションエラーはインラインで各フィールド下部に表示。サーバーエラーはフォーム上部のエラーバナーに表示

### セキュリティ
- **認証方式**: 社内 SSO(SAML 2.0)。フロントエンドは SSO プロキシを経由し、バックエンドは `X-Employee-Id` ヘッダーから社員 ID を取得
- **認可方式**: 予約の変更・取消は `booking.employeeId === requestEmployeeId` をサービス層でチェック。他人の予約は 403 Forbidden を返す
- **CSP / ヘッダー**: Nginx で `X-Content-Type-Options: nosniff` / `X-Frame-Options: DENY` / `Referrer-Policy: strict-origin-when-cross-origin` を付与
- **暗号化**: HTTPS のみ(社内 CA 証明書)。DB 接続は TLS

## 非機能要件

### パフォーマンス
| 指標 | 目標値 | 測定方法 |
|------|--------|---------|
| API レスポンスタイム(p95) | 200ms 以内 | k6 負荷テスト(同時 50 接続) |
| 初回ページロード(LCP) | 2.5s 以内 | Lighthouse CI |
| カレンダー日付切り替えの体感速度 | キャッシュヒット時即時、ミス時 300ms 以内 | 手動計測 |

### スケーラビリティ
- **想定ユーザー数**: 社員 300 名、同時接続 30 名を上限に設計
- **拡張方針**: 現状は 1 Pod で運用。会議室数・社員数が増えた場合は Pod 数を増やす

### 可用性
- **目標稼働率**: 95%(社内ツールのため SLA は厳格には求められない。平日業務時間帯の稼働を優先)
- **障害時の復旧方針**: Pod 障害時は Kubernetes の自動再起動に依存。DB 障害時はアプリが 503 を返し、数分以内に復旧を目指す

### 監視・ログ
- **監視ツール**: 社内 Prometheus + Grafana(既存インフラを流用)
- **ログレベル**: INFO(通常操作) / WARN(バリデーションエラー) / ERROR(サーバーエラー・DB 接続失敗)
- **ログ保持期間**: 30 日(社内ログ管理ポリシー)
- **アラート条件**: API エラーレート 5% 超 / DB 接続失敗が 3 回連続 → Slack の社内 ops チャンネルに通知

## 質疑応答ログ

### Q-01 — ダブルブッキング防止は DB レベルで保証するか、アプリ層でのみ行うか?
- **回答**(人間の回答を AI が記入):
  > DB レベルでも保証してほしい。アプリ層だけだと同時アクセスで抜ける可能性が怖い。
- **確定**(AI 記入):
  > PostgreSQL の `SELECT FOR UPDATE` を使ったトランザクション内での排他ロックを採用する。アプリ層のチェックは先行検証(ユーザーへの早期エラー表示)として残し、DB レベルを最終防衛線とする。

## AI が独自に決めたこと と 理由

### D-01 — バックエンドを Node.js + Express にする
- **理由**: チームに既存知見があり、Prisma との TypeScript ファーストな組み合わせが相性よい。将来 Hono への移行コストも低い。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-02 — React Query をサーバー状態管理に使う
- **理由**: 予約一覧・カレンダーデータはサーバーの最新状態を反映する必要があり、キャッシュ+再検証のパターンが自然にフィットする。Redux 等のグローバルストアはオーバーキル。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-03 — 会議室 ID と表示日付を URL クエリパラメータに持つ
- **理由**: ブックマーク・ブラウザバックで同じカレンダーに戻れるようにするため。社内 Slack でリンクを共有したときにも有効。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## 棄却した案

### R-01 — GraphQL の採用
- **棄却理由**: 画面数が少なく、REST API + React Query で十分。GraphQL を導入すると N+1 対策・スキーマ設計・Codegen 設定のコストが見合わない。

### R-02 — リアルタイム WebSocket(予約状況の自動更新)
- **棄却理由**: 社員 300 名規模のツールで常時接続を維持するのはオーバーエンジニアリング。ユーザーが日付を切り替えるタイミングで再フェッチすれば実用上十分。
