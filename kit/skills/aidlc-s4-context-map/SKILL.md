# AI-DLC S4: コンテキストマップ

## あなたの役割

あなたは引き続き **ソフトウェアアーキテクト** です(S3 と同じ役割で連続性を保つ)。

S3 で設計した Unit 群の **Unit 間依存方向** を、**1 枚の Mermaid 図に集約して全員の共通言語にする**のが責務。読み手は「**どの Unit から並行開発に着手できるか**」(依存の浅い Unit から先に作れる)を 1 枚で判断できる。

## PDF 原典(P.6 / S4)

PDF P.6「S4 — Unit 間関係の可視化」は以下:

| 項目 | 内容 |
|------|------|
| 入力 | Unit 一覧、I/F 定義 |
| 出力 | Mermaid 形式のコンテキストマップ |
| 趣旨 | Unit 間の依存・境界を図で可視化し、**チーム全員が同じ地図を持つ** |

**PDF には永続化(DB)・外部サービス・プロトコル境界・物理デプロイ境界の話は一切書かれていない**。これらは S5(ドメインモデル)/ S7(統合)の領域。S4 で描き込むと役割が混線する。

## いまどこにいるか

- フェーズ: **Inception** の S4(Inception 最終ステップ)
- 前: `aidlc-docs/{version}/s3/`(Unit 一覧と I/F 定義)
- 後: S5(ドメインモデル)= Construction フェーズの入口
- バージョン: `aidlc-docs/{version}/` が現在のサイクル。`{version}` は作業中バージョン(v0.0.2 等)に置換する。`aidlc-docs/brief.md` のみ全版共通(ルート)。

## 入出力と完了条件

| 項目 | 内容 |
|------|------|
| 入力 | `aidlc-docs/{version}/s3/`(`index.md` と `unit-NN-*.md` 群) |
| 出力 | `aidlc-docs/{version}/s4-context-map.md`(Mermaid 図入り) |
| 完了条件 | (1) S3 で定義した **全 Unit** がノードとして図に登場 / (2) **Unit 間の依存方向**(誰が誰を呼ぶか・必要とするか)が矢印で明示 / (3) **循環依存が無い**(あれば S3 に戻る) / (4) 1 枚で全体像が読める(分割しない) / (5) **物理アーキ要素(DB / 外部サービス / プロトコル境界 / Docker network 等)を描き込んでいない**(S5 / S7 の領域に侵食しない) / (6) **図そのものが実装順を表現している**(=Phase レイアウトで subgraph を縦に並べ、上から下に読めば着手順が分かる) |

## 進め方

0. **最初に必ず**:
   - `s3/index.md` と `s3/unit-NN-*.md` 群を読む。Unit と I/F 定義を全件把握。
   - `s4-context-map.md` が既にあれば「前サイクルからの引き継ぎ」を最優先で反映。
   - 無ければ下のテンプレで新規作成。**以降の質疑応答は md 上で行う**: AI が `### Q-NN` を md に追記 → **ユーザーが IDE で md を直接編集して `回答` を書き込む** → AI が次のやり取りで `確定` を埋める。

1. AI が Mermaid `flowchart` で初版を描く。記号運用(矢印の意味、subgraph の使い方)は md 内に **凡例** として書く。

2. AI が独自に決めた「境界の引き方」「subgraph のくくり」は **「AI が独自に決めたこと と 理由」に `### D-NN` で必ず追記**。ユーザーは `判断` 行で `承認 / 上書き / 保留` を選び、上書きするなら `上書き内容` に直接書く。

3. ユーザーと一緒に「この図を見て並行開発者が誤読しないか」を確認。曖昧な境界は注記で補強。

4. 完了条件 4 つが全て埋まったら `ステータス: 確定` にして S5 を案内。

## 成果物 md テンプレート

```markdown
# S4 — コンテキストマップ

## メタ
- 工程: S4 (Context Map)
- 役割: ソフトウェアアーキテクト
- ステータス: 進行中 | レビュー待ち | 確定
- 入力参照: [s3/index.md](./s3/index.md)
- 作成日: YYYY-MM-DD
- 更新日: YYYY-MM-DD

## 全体図 (Unit 間依存 DAG / **Phase レイアウト**)

\`\`\`mermaid
flowchart TB
  subgraph P1["Phase 1 (leaf / 並行可)"]
    direction LR
    U03[Unit-03<br/>{責務 1 行}]
    U04[Unit-04<br/>{責務 1 行}]
  end

  subgraph P2["Phase 2"]
    U02[Unit-02<br/>{責務 1 行}]
  end

  subgraph P3["Phase 3"]
    U01[Unit-01<br/>{責務 1 行}]
  end

  U01 --> U02
  U01 --> U03
  U02 --> U04
\`\`\`

**読み方**:
- 矢印は **依存方向**(`A --> B` = A は B を呼ぶ / A は B に依存する)。
- **上から下に読めば着手順**(上のほうの Phase ほど先に作る)。Phase 内の Unit は並行に着手できる。
- **矢印は下から上または同 Phase 内向き**(=依存先は必ず自分より上の Phase = 先に作られている)。**矢印が上から下に伸びていたら循環の疑い**(S3 に戻る判断材料)。

## 凡例
- **角括弧 `[X]`**: Unit(S3 で定義した自前 Unit のみ)
- **実線矢印 `-->`**: 依存方向(`A --> B` は「A は B を呼ぶ / A は B が無いと動かない」)
- **点線矢印 `-.->`**: 弱い依存(イベント駆動・非同期 / 例: A が B にメッセージを投げて以降は知らない)
- **subgraph**: **S4 の正規用途は「Phase = 実装順の段」を表現すること**(Phase 1 = leaf = 最初に着手、Phase N = 最後に着手)。図そのものが実装順の地図になる。**物理境界(端末 vs サーバ等)を表現するための subgraph は使わない**(必要なら各 Unit の責務 1 行に書く)。プロトコル境界・Docker network・認証境界・テスト戦略境界などの実装詳細を理由に subgraph を切らない。

**意図的に使わない記号**:
- 円柱 `[(X)]` 永続化、六角 `{{X}}` 外部サービス → S4 では描かない(S5 / S7 の領域)
- プロトコル種別を矢印太さで区別(`==>` 等)→ S4 では描かない(プロトコルは S3 の I/F 定義と S7 統合の話)

## 依存方向の根拠
| 依存(A → B) | 根拠 (なぜ A は B に依存するか) |
|--------------|-------------------------------|

## 並行開発の着手順(図中の Phase subgraph と一対一対応)

| Phase | 着手可能な Unit | 理由 |
|-------|----------------|------|
| Phase 1(leaf) | Unit-XX, Unit-YY | 他 Unit に依存しない |
| Phase 2 | Unit-ZZ | Phase 1 が揃えば作れる |
| ... | | |

**重要**: この表は **図中の Phase subgraph と一対一対応** している(=表が Phase の根拠説明、図が Phase の視覚化)。Phase 数や所属 Unit を変更したら必ず両方を同期する。

## 読み手別の見方
- **エンジニア**: 自分の担当 Unit がどの Unit に依存しているか(矢印の先)を見て、先にスタブを用意すべき相手を把握する。詳細 I/F は [s3/](./s3/) の各 Unit ファイルを参照。
- **PM**: 依存の浅い Unit(leaf)から並行に着手できることを確認 / 着手順テーブルで進捗計画に使う。

## 質疑応答ログ
```

## やってはいけないこと

- `s3/` を読まずに図を描く(Unit と I/F 定義の根拠が失われる)
- 1 枚に収まらないからと言って図を分割する(全体像を持つ目的に反する)
- 凡例なしで Mermaid を書く(矢印の意味が読み手によってずれる)
- Mermaid 以外の独自記法を採用する(チーム共通言語にするため Mermaid 固定)
- 実装詳細(クラス名、テーブル名など)を図に描き込む(S5 以降の領域)
- **物理アーキ要素を描き込む**(以下は S5 ドメイン設計 / S7 統合の領域 / S4 では一切描かない):
  - 永続化(`[(DB)]` 円柱 / Postgres / Object Storage / Job Queue テーブル など)
  - 外部サービス(`{{X}}` 六角 / APNs / LLM provider / 認証プロバイダ など)
  - プロトコル種別の区別(REST / HTTP MCP / gRPC / WebSocket / subprocess 等を矢印太さで分類しない)
  - Docker network / プロセス境界 / コンテナ単位 / デプロイ単位 を subgraph で囲わない
  - 認証方式 / API キー / 環境変数 等を矢印ラベルに書かない
- **Phase(実装順)以外の理由で subgraph を切る**(プロトコル境界・テスト戦略境界・物理デプロイ境界・Docker network はすべて NG / **subgraph は Phase 表現のためだけに使う**)
- **v0.2 切替計画や将来計画を図に描き込む**(現在の依存方向グラフのみ / 将来計画は S3 の `index.md` か S7 引き継ぎで書く)
- **「順序関係」を呼び出しシーケンス図(時系列フロー)として描く**(S4 は **依存方向 DAG** であって sequence diagram ではない / シーケンスを描きたければ別途 `sequenceDiagram` を `s3/` の該当 Unit ファイルに置く)

## やり直しの判断

- **S3 に戻る**: 図を描こうとして「Unit の依存が循環している」「Unit の責務が重複している」と気づいた
- **S2/S1 に戻る**: Unit 設計の前提となる US/画面そのものが破綻していた(まれ)
- **このまま S4 で書き直す**:
  - 図は描けたが読み手が誤読する(凡例強化・subgraph 再構成で対処)
  - **物理アーキ要素を描き込んでしまった**(DB / 外部サービス / プロトコル境界 / Docker network 等)→ それらを削除して Unit ノード + 依存矢印だけに戻す

戻る際は対象 md 末尾の **前サイクルからの引き継ぎ** に 1〜3 行で書く(全量コピーしない):

- 何が漏れていたか
- 暫定の解決方針
- 棄却した案とその理由
