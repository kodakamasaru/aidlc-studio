# AI-DLC S5: Work Units (Unit 分割 + 依存マップ)

## あなたの役割

あなたは **ソフトウェアアーキテクト** です。

US 群を「並行開発できる Unit」に分解し、各 Unit の責務境界と I/F 契約を設計すると同時に、Unit 間依存方向を DAG で可視化して実装着手順を定める責任者。**Unit の妥当性はエンジニアであるあなたが最終判断する**(PDF 強調)。

AI 提案は叩き台。Unit の粒度、責務、依存方向を 1 つ 1 つ吟味する。「AI がそう言ったから」で採用しない。

## いまどこにいるか

- フェーズ: **Build** の S5(Build PhaseGroup の起点)
- 前: `aidlc-docs/{version}/s1/`(確定 US 群) + `aidlc-docs/{version}/s2/`(画面/フロー) + `aidlc-docs/{version}/s3/`(UI デザイン, 存在する場合) + `aidlc-docs/{version}/s4-tech-spec.md`(技術仕様, 存在する場合) + `aidlc-docs/brief.md`
- 後: S6(ドメインモデル)の入力になる
- バージョン: `aidlc-docs/{version}/` が現在のサイクル。`{version}` は作業中バージョン(v0.0.2 等)に置換する。`aidlc-docs/brief.md` のみ全版共通(ルート)。

## 入出力と完了条件

| 項目 | 内容 |
|------|------|
| 入力 | `aidlc-docs/{version}/s1/` + `aidlc-docs/{version}/s2/` + `aidlc-docs/{version}/s4-tech-spec.md`(存在する場合) + `aidlc-docs/brief.md` + アーキテクチャ前提情報 |
| 出力 | `aidlc-docs/{version}/s5/` 配下に `index.md` + `unit-NN-{slug}.md` 群 |
| 完了条件 | (1) 全 US がいずれかの Unit に割り当て / (2) 各 Unit に責務・所属 US・I/F 定義案がある / (3) Unit 間の依存方向が一方向 or 明示された境界 / (4) 依存 DAG に循環が無い / (5) 依存 DAG に物理アーキ要素(DB / 外部サービス / プロトコル境界等)を描き込んでいない / (6) Phase subgraph と着手順テーブルが一対一対応 / (7) エンジニア(=ユーザー本人)が「これで並行開発できる」と納得 |

## 進め方

0. **最初に必ず**:
   - `s1/index.md` と `s2/index.md`、および主要な単位ファイルを読む。
   - `s4-tech-spec.md` が存在すれば読み、技術前提を把握する。
   - `aidlc-docs/{version}/s5/` が既にあれば `index.md` の引き継ぎセクションと各 `unit-NN-*.md` を確認。
   - 無ければ `aidlc-docs/{version}/s5/` を作り、下のテンプレで `index.md` を新規作成。
   - **以降の質疑応答は md 上で行う**: AI が `### Q-NN` を md に追記 → **ユーザーが IDE で md を直接編集して `回答` を書き込む** → AI が次のやり取りで `確定` を埋める。
   - **質問の所属**: アーキ全体・I/F 方針・Unit 横断・依存マップ全体は `index.md`、個別 Unit の議論は該当 `unit-NN-*.md`。

1. **I/F 決定方針を最初に決める**(PDF 提示の選択、`index.md` に記録):
   - (a) **人間が事前に決める**: ユーザーから I/F を聞き出して埋める
   - (b) **AI に事前調査させる**: 既存コード/類似 OSS/ドキュメントを AI が調べて I/F 案を出す

2. **アーキテクチャ前提を確認**(PDF 警告: 実 PJ と乖離すると AI が的外れな質問を返す)。`s4-tech-spec.md` があればそこから技術前提を継承。

3. AI が Unit 候補(3〜8 程度)を提案。**1 Unit 1 ファイル**で起こす。
   - ファイル名: `unit-NN-{kebab-slug}.md`
   - 各 Unit に「責務・所属 US(`../s1/us-NN-*.md` リンク)・外部依存・I/F 定義案」を埋める。
   - `index.md` の「Unit 一覧」にリンクを追加。

4. AI が Mermaid `flowchart` で Unit 間依存 DAG を `index.md` に描く。記号運用(矢印の意味、Phase subgraph の使い方)は `index.md` 内の **凡例** に書く。**Phase subgraph は実装着手順の段を表現するためだけに使う**(物理境界には使わない)。

5. AI が独自に決めた粒度・命名・分割・境界の引き方・Phase 構成は **該当 md の「AI が独自に決めたこと と 理由」に `### D-NN` で追記**。ユーザーは `判断` で `承認 / 上書き / 保留` を選び、上書きするなら `上書き内容` に直接書く。

6. ユーザーと一緒に Unit 単位の妥当性を 1 件ずつ吟味。納得できない Unit は再分割 or 統合。依存 DAG も「この図を見て並行開発者が誤読しないか」を確認。

7. 完了条件 7 つが全て埋まったら `index.md` のステータスを `確定` にして S6 を案内。

## 依存 DAG (Unit 間依存方向 / Phase レイアウト)

\`\`\`mermaid
flowchart TB
  subgraph P1["Phase 1 (leaf / 並行可)"]
    direction LR
    U03[Unit-03<br/>{責務 1 行}]
    U04[Unit-04<br/>{責務 1 行}]
  end

  subgraph P2["Phase 2"]
    U02[Unit-02<br/>{責務 1 行}]
  end

  subgraph P3["Phase 3"]
    U01[Unit-01<br/>{責務 1 行}]
  end

  U01 --> U02
  U01 --> U03
  U02 --> U04
\`\`\`

**読み方**:
- 矢印は **依存方向**(`A --> B` = A は B を呼ぶ / A は B に依存する)。
- **上から下に読めば着手順**(上のほうの Phase ほど先に作る)。Phase 内の Unit は並行に着手できる。
- **矢印は下から上または同 Phase 内向き**(=依存先は必ず自分より上の Phase = 先に作られている)。**矢印が上から下に伸びていたら循環の疑い**(Unit 分割の見直しが必要)。

## 凡例
- **角括弧 `[X]`**: Unit(本ステップで定義した自前 Unit のみ)
- **実線矢印 `-->`**: 依存方向(`A --> B` は「A は B を呼ぶ / A は B が無いと動かない」)
- **点線矢印 `-.->`**: 弱い依存(イベント駆動・非同期 / 例: A が B にメッセージを投げて以降は知らない)
- **subgraph**: **Phase = 実装順の段**を表現する(Phase 1 = leaf = 最初に着手、Phase N = 最後に着手)。図そのものが実装順の地図になる。**物理境界(端末 vs サーバ等)を表現するための subgraph は使わない**(必要なら各 Unit の責務 1 行に書く)。プロトコル境界・Docker network・認証境界・テスト戦略境界などの実装詳細を理由に subgraph を切らない。

**意図的に使わない記号**:
- 円柱 `[(X)]` 永続化、六角 `{{X}}` 外部サービス → S5 では描かない(S6 / S8 の領域)
- プロトコル種別を矢印太さで区別(`==>` 等)→ S5 では描かない(プロトコルは I/F 定義と S8 統合の話)

## 着手順テーブル (Phase subgraph と一対一対応)

| Phase | 着手可能な Unit | 理由 |
|-------|----------------|------|
| Phase 1(leaf) | Unit-XX, Unit-YY | 他 Unit に依存しない |
| Phase 2 | Unit-ZZ | Phase 1 が揃えば作れる |
| ... | | |

**重要**: この表は **図中の Phase subgraph と一対一対応** している(=表が Phase の根拠説明、図が Phase の視覚化)。Phase 数や所属 Unit を変更したら必ず両方を同期する。

## 依存方向の根拠
| 依存(A → B) | 根拠 (なぜ A は B に依存するか) |
|--------------|-------------------------------|

## 読み手別の見方
- **エンジニア**: 自分の担当 Unit がどの Unit に依存しているか(矢印の先)を見て、先にスタブを用意すべき相手を把握する。詳細 I/F は各 Unit ファイルを参照。
- **PM**: 依存の浅い Unit(leaf)から並行に着手できることを確認 / 着手順テーブルで進捗計画に使う。

## やってはいけないこと

- AI 提案の Unit 構成を人間レビューなしで「確定」する(PDF 強調: Unit 妥当性はエンジニアが判断)
- US が割り当てられていない Unit、Unit に割り当てられていない US を残す
- I/F 定義を「あとで」と空欄のまま S6 に進める(S8 で必ず突き合わせるため)
- 双方向依存(循環)を残す。境界を引き直すか、依存方向を片方向に整理する
- スタック情報を実 PJ から乖離させたまま進める(S6 で AI が的外れな質問を返す原因になる)
- Unit 横断の議論を個別 Unit ファイルに書く(`index.md` に書く)
- Unit ファイルを `index.md` に登録せず孤立させる
- `s1/` `s2/` を読まずに Unit 分割や依存マップを描く(根拠が失われる)
- 凡例なしで Mermaid を書く(矢印の意味が読み手によってずれる)
- Mermaid 以外の独自記法を採用する(チーム共通言語にするため Mermaid 固定)
- 実装詳細(クラス名、テーブル名など)を依存マップに描き込む(S6 以降の領域)
- **物理アーキ要素を描き込む**(以下は S6 ドメイン設計 / S8 統合の領域 / S5 では一切描かない):
  - 永続化(`[(DB)]` 円柱 / Postgres / Object Storage / Job Queue テーブル など)
  - 外部サービス(`{{X}}` 六角 / APNs / LLM provider / 認証プロバイダ など)
  - プロトコル種別の区別(REST / HTTP MCP / gRPC / WebSocket / subprocess 等を矢印太さで分類しない)
  - Docker network / プロセス境界 / コンテナ単位 / デプロイ単位 を subgraph で囲わない
  - 認証方式 / API キー / 環境変数 等を矢印ラベルに書かない
- **Phase(実装順)以外の理由で subgraph を切る**(プロトコル境界・テスト戦略境界・物理デプロイ境界・Docker network はすべて NG / **subgraph は Phase 表現のためだけに使う**)
- **将来計画を依存マップに描き込む**(現在の依存方向グラフのみ / 将来計画は `index.md` の別セクションか S8 引き継ぎで書く)
- **「順序関係」を呼び出しシーケンス図(時系列フロー)として描く**(S5 は **依存方向 DAG** であって sequence diagram ではない / シーケンスを描きたければ別途 `sequenceDiagram` を該当 Unit ファイルに置く)

## やり直しの判断

- **S1 に戻る**: Unit を切ろうとして「US の粒度がバラバラ」「US が抜けている」と気づいた
- **S2 に戻る**: 画面/フローと Unit の境界がどうしても合わない(US が画面の上に乗っていない可能性)
- **このまま S5 で書き直す**: Unit 数が多すぎる/少なすぎる、依存方向が解消できない、物理アーキ要素を描き込んでしまった(それらを削除して Unit ノード + 依存矢印だけに戻す)

戻る際は `index.md` の **前サイクルからの引き継ぎ** に 1〜3 行で書く(全量コピーしない):

- 何が漏れていたか
- 暂定の解決方針
- 棄却した案とその理由
