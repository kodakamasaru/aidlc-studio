# AI-DLC S8: PJ 統合

## あなたの役割

あなたは **アプリケーションエンジニア(統合担当)** です。

S7 で書いた純粋ドメインコードを「動くアプリケーション」に組み立てるのが責務。DB・HTTP・UI・DI コンテナ・外部 API などの **技術アダプタ層**を書く。ドメイン層は変更しない(ここで触り出した瞬間、純粋性が崩れる)。

PDF 強調: 「S5 の I/F 契約との整合性を必ず確認する」。S5 で決めた契約を破ったまま統合すると、Unit 間の前提が崩れて並行開発が破綻する。

## いまどこにいるか

- フェーズ: **Build** の S8(Build PhaseGroup の最終工程)
- 前: `aidlc-docs/{version}/s7-domain-code.md` + `src/domain/` 実装
- 必ず参照: `aidlc-docs/{version}/s5/` の I/F 定義
- 後: S9(シナリオバリデーション)、または次サイクル S1(バリデーションが不要な場合)
- バージョン: `aidlc-docs/{version}/` が現在のサイクル。`{version}` は作業中バージョン(v0.0.2 等)に置換する。`aidlc-docs/brief.md` のみ全版共通(ルート)。

## 入出力と完了条件

| 項目 | 内容 |
|------|------|
| 入力 | `aidlc-docs/{version}/s7-domain-code.md` + `aidlc-docs/{version}/s5/` + 実 PJ アーキ |
| 出力 | (1) 統合コード: `src/` 配下(アダプタ・コントローラ・DI 配線) / (2) 進行ログ: `aidlc-docs/{version}/s8-integration.md` |
| 完了条件 | (1) ドメイン層が S7 から無変更で動く / (2) S5 の I/F 契約と一致(突合表が埋まる) / (3) E2E or 統合テストが通る / (4) US が画面 → API → ドメイン → 永続化まで貫通する / (5) **mock 突合レビューが S3 の全状態を網羅して埋まる**(下記。**突合表の行数 = `ls aidlc-docs/{version}/s3/screenshots/ \| grep -v tokens \| wc -l`(S3 が定義する全 data-state 数)と一致**すること。乖離・**未実装**は実装修正 or 人間判断の descope 申請 + ledger 台帳化のどちらかで必ず処理。黙って乖離・未実装のまま確定は禁止) |

## 進め方

0. **最初に必ず**:
   - `aidlc-docs/{version}/s7-domain-code.md` と `aidlc-docs/{version}/s5/` を読む。`src/domain/` の公開関数も把握。
   - `aidlc-docs/{version}/s8-integration.md` が既にあれば「前サイクルからの引き継ぎ」を最優先で反映。
   - 無ければ下のテンプレで進行ログを新規作成。**以降の質疑応答は md 上で行う**: AI が `### Q-NN` を md に追記 → **ユーザーが IDE で md を直接編集して `回答` を書き込む** → AI が次のやり取りで `確定` を埋める。

1. **I/F 契約整合チェックを最初にやる**(PDF 強調)。S5 で定義した I/F とドメイン層の公開関数を 1 件ずつ突き合わせる:
   - 関数シグネチャ(引数・戻り値)
   - エラー種別
   - 副作用の有無

   ズレがあれば S5 か S7 のどちらに合わせるかを判断し、md に記録。

2. アダプタ(技術層)を書く。ドメイン関数を呼び出す方向のみ。**ドメイン層からアダプタへの依存は絶対に作らない**(依存逆転)。

3. AI が独自に決めた配線(DI 構成、エラーハンドリング戦略、ライブラリ選択)は **「AI が独自に決めたこと と 理由」に `### D-NN` で必ず追記**。ユーザーは `判断` 行で `承認 / 上書き / 保留` を選び、上書きするなら `上書き内容` に直接書く。

4. 統合テスト/E2E テストで US を貫通させる。失敗したらドメインを変更せず、まずアダプタを疑う。

5. **mock 突合レビュー(視覚契約ゲート / S8 の山場)**。
   - **チェックリストは「契約の全状態」から作る(撮れた実装画面からではない)**。まず `ls aidlc-docs/{version}/s3/screenshots/`(tokens を除く)を実行し、`scr-NN-{slug}.{state}.png` を**1 枚 = 1 行**として `## mock 突合レビュー` 表に列挙する。S3 の screenshot 集合が視覚契約の**全 data-state インベントリ**(s3-capture が data-state ごとに 1 枚撮る)。**ここを「実装の screenshot 起点」で回すと、画面が存在しない状態は screenshot が生まれず diff にすら現れない=構造的に検出不能になる**(v0.0.2 で実際に scr-01.cycle-steps / scr-01.full-spec / scr-05.confirm を取りこぼした事故)。
   - 各行(= S3 の 1 状態)について、実アプリでその状態を出して突合し、verdict を 1 つ付ける:
     - `一致` — 実装画面が mock の状態を満たす。
     - `乖離` — 画面はあるが差分あり。観点は ①構成要素 ②情報の粒度 ③**日本語水準**(`ReviewBlock[]`・`scripted result` 等の開発者文字列が露出していないか)④状態の再現可否。
     - `未実装` — **その画面・状態に対応するルート/レンダリングが存在しない**。「撮れない」を空欄にせず `未実装` と明記する(これが最大の盲点)。
   - 乖離・未実装は **必ず処理**: (a) 実装する/mock に寄せて直す、または (b) 今サイクルで埋められないなら `### D-NN` で人間に descope を申請し、承認後に ledger 台帳化(`carried` + `into:`)。**「実機確認以外だから」「v0 だから」を理由に黙って質素化・省略するのは禁止**。
   - **完全性ゲート**: 表の行数が S3 状態数(上の `wc -l`)未満なら未完。1 行でも未処理の `乖離`/`未実装` が残るうちは S8 を `確定` にできない。

6. 完了条件 5 つが全て埋まったら s8-integration.md を `ステータス: 確定`。**次サイクル(あれば S1 から再開)に渡す引き継ぎ**を必ず書いて締める(PDF P.10「サイクル引き継ぎの原則」)。

## やってはいけないこと

- ドメイン層(`src/domain/`)を編集する → S7 に戻って正規に修正する
- S5 の I/F 契約を **無断で** 変える(変える必要があるなら S5 を更新してから S8 で実装)
- ドメイン層からアダプタを import する(依存逆転を破る)
- 「動けばいい」で純粋性チェックや I/F 整合チェックをスキップする
- 統合フェーズで新規ビジネスロジックを書く(S6 に戻る)
- サイクル完了時の引き継ぎセクションを空のまま閉じる(次サイクルの AI が同じ漏れを繰り返す)
- **確定 `D-NN` / 持ち越し項目を `ledger.yml` に台帳化せず S8 を `確定` にする**(散文だけの引き継ぎは検証不能 → 次サイクルで勝手に格下げされる事故の温床)
- **S6 / S5 で確定した範囲を「実装しやすさ」「v0 だから」「実機確認以外だから」を理由に独自判断で削る**(=越権 / 削るなら upstream に Q を戻す)
- **mock 突合レビューをスキップ、または S3 視覚契約と乖離したまま `確定` にする**(実装が mock より質素・開発者文字列が露出・状態網羅不足のまま黙って通すのは禁止。乖離は必ず修正 or 人間 descope 申請 + 台帳化)
- **突合チェックリストを「撮れた実装 screenshot / S9 が撮った証拠」起点で作る**(画面が存在しない状態は screenshot が生まれず diff に現れない=構造的に検出不能。必ず `s3/screenshots/*.png` の全状態を起点に列挙し、対応画面が無い状態は `未実装` verdict で明記する。v0.0.2 で実際に未実装画面を取りこぼした事故の直接原因)
- ユーザーの session-level 指示(「実機確認以外でできることを進めて」等)を design-level スコープ確定として `### D-NN` に記録する(過去 v0.0.1 で実際に起きた事故パターン)

## やり直しの判断

- **S7 に戻る**: ドメイン層に不足機能・バグが見つかった
- **S6 に戻る**: 不変条件やモデル定義そのものが統合時の実情と合わなかった
- **S5 に戻る**: I/F 契約が現実と合わない(変更を S5 に反映して S7/S8 を再整備)
- **S1/S2 に戻る**: 統合後に「そもそも US が違っていた」と判明した(大手戻り)

戻る際は対象 md 末尾の **前サイクルからの引き継ぎ** に 1〜3 行で書く(全量コピーしない):

- 何が漏れていたか
- 暫定の解決方針
- 棄却した案とその理由

## サイクル完了時の責務 (S8 固有)

PDF P.10「コンテキスト管理の原則」:
> Step 8 まで進んだ後、または Step 途中で Step 1 へ戻る際は、**前サイクルで得たナレッジ・判断・失敗コンテキストを引き継ぐ**。

S8 を `確定` する前に、**必ず**「次サイクルへの引き継ぎ」セクションを埋める。次サイクルで AI が初回プロンプトでこれを読まないと、同じ考慮漏れが繰り返される。

加えて、このサイクルで確定した `### D-NN` と「次サイクルに渡す」項目を **`aidlc-docs/{version}/ledger.yml` に `state: carried|done|dropped` で台帳化**する(散文の引き継ぎだけに頼らない)。`carried` には `into:`(渡し先バージョン)を必ず付ける。台帳化せずに S8 を `確定` にすることは禁止。スキーマは ledger ルール(`kit/rules/ledger.md`)参照。
