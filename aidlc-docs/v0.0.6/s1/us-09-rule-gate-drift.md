## 3 視点

### なぜするか (Why)
AI-DLC の「あるべき挙動」が ① kit 散文 ② 機械ゲート(コード)③ データモデル/UI の 3 箇所に別符号化され、単一正本が無い。kit を直しても機械ゲートは追従せず(逆も)= ドリフトする。precision-first 原則(存在/形式/連結=機械ゲート / 質=evaluator / 最終確信=人間 / 狙い=平易 intent)に従い、「機械強制を謳う binding ルールが gate + test を名指ししているか」を機械検査し、ドリフトを赤にする。あわせて単一正本の「橋の型」を 1 本実証して、以後の不変条件を二重符号化せず型に寄せられるようにする。

### UX へのインパクト
(主にメンテナンス品質の不可視な担保。)方法論を更新したのにプラットフォームが追従していない不整合を、人間がレビューで見つける前に CI/gate が赤で止める。ユーザーは「ルールは直したのに挙動が変わっていない」事故に遭わなくなる。

### 受け入れ条件 (AC)
- `probe:rules` を拡張し、**「機械強制を謳う binding ルール ↔ gate id ↔ test」の連結を機械検査**する。連結が欠ければ **exit 1(赤)**(設計 BACKLOG §K (2) / (3a))。
- ゲートが見るのは**連結の有無まで**(precision-first: 存在=ゲート)。中身の質は evaluator、最終確信は人間に委ねる(原則どおりの振り分け)。
- **橋の型を 1 本実証**: `requiresLiveEvidence`(既に step-contracts → 散文役割 + ゲートの両駆動)に倣い、**既存の不変条件 1〜2 件を step-contracts 駆動へ移行**して型を確立する(BACKLOG §K (3b))。
- 新しい方法論不変条件は step-contracts(型付き単一正本)に宣言し、散文もゲートもそこから導出する規律を operating-model に記す((3a) / 濃い規則文でなく連結ゲート + 平易 intent で効かせる)。
