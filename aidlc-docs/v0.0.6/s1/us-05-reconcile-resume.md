## 3 視点

### なぜするか (Why)
自己復帰の正体は keep-alive ではなく「起動毎に DB と実プロセスを突合して正しい続きへ戻す」durable なステートマシン(設計 §2)。アプリ/backend がクラッシュ・再起動しても不整合を起こさず続きに戻れねば、放置自走は信用できない。現状の reconcile は孤児を stall 化するだけで、resume 優先 + 自動 retry になっていない(設計 §11)。あわせて O3(揮発しない実 session の `--resume` 継続)が live 未実証で、本経路で実機実証する。

### UX へのインパクト
PC を再起動しても、アプリが落ちても、立ち上げ直せば走っていたタスクが正しい続きから再開する。ユーザーは「途中だったあれ、どうなった?」を気にしなくてよい。中断は無かったかのように進む。

### 受け入れ条件 (AC)
- 起動時 reconcile が、DB の `running` と実プロセスを突合し(稼働台帳 US-08 の pid を読む)、孤児 run を検知 → **resume 優先**(session_id があれば同一文脈継続)、無ければ **idempotent な re-run**(設計 §5/§8)。
- backend を **mid-run で kill → 再起動**しても、起動後に全 run が**正しい続き**に戻る(不整合・取りこぼしゼロ / 設計 §12-1)。
- 二重起動しない(冪等 / desired vs actual の再導出)。
- **O3 実証**: 揮発しない実 session を用いた `--resume` 継続を**実 claude の実機シナリオ**で通す(reconcile-resume 経路で session が継続し、同一文脈で step が完走する dossier を残す)。

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — 復旧が戻すのは「パイプライン上の位置」であって「同じバイト列」ではない
- **理由**: 設計 §5。claude は非決定的なので resume が忠実、re-run は新試行。human-gate 前なので新試行で問題ない。resume を優先(session_id 永続済み)、resume 不能時のみ idempotent re-run。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-02 — O3 を本 US に内包(独立 US にしない)
- **理由**: index D-03 / ledger の O3 carry 方針。resume 経路の実機シナリオが O3 の実証要件(揮発しない実 session の --resume 継続)を完全に満たすため、独立 US 化は検証重複。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## この US 固有の 棄却した案

### R-01 — 孤児 run を常に re-run(resume を使わない)
- **棄却理由**: 設計 §5「resume を優先」。re-run は文脈を失い新試行になる。session_id があるなら resume で文脈継続する方が忠実。
