## 3 視点

### なぜするか (Why)
プロダクトの価値の源泉は「放置度(hands-off 度)」。今は人間が 1 フェーズずつ手動で起動している。タスクを多数積んでおけば、人間が放置していても、AI が人間の判断が要る所まで勝手に進む — これが無いと「ボードに積むだけ」が成立せず、製品の魂(Human Inbox に判断だけが上がってくる)が回らない。

### UX へのインパクト
ユーザーはやりたいことをボードに積んで離席できる。戻ってくると、AI が止まったのは「自分の判断が要る所(要件 / 画面 / レビュー / 改善)」だけで、技術的に進められる所は全部進んでいる。同時に走るタスク数は上限内に保たれ、暴走しない。

### 受け入れ条件 (AC)
- pending なタスク群から、**依存 DAG・human-gate・並列上限 N** を見て起動可能なものだけを起動する。
- 並列起動数が**常に N 以下**(実機シナリオで確認 / 設計 §12-5)。
- human-gate(要件 / 画面 / 視覚レビュー / 受け入れ / 改善)に当たった step は run を「待ち」状態で**永続 parking** し、人間が後で(数日後でも)答えられる。
- human-gate 不要で完了した step の後は、**次の eligible step を自動で起動**する(逐次自走)。
- 起動毎に desired(DB のあるべき)vs actual(実プロセス)を再導出して(再)起動し、**二重起動しない**(冪等)。
- in-memory に実行状態を置かない(DB が唯一の真実 / 再起動で状態が消えない)。

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — 並列は「プロセス並列」であって worktree 並列ではない
- **理由**: worktree 複数による真の並行サイクル(N>1 を別 worktree で / 書込競合分離)は v0.0.7(AUTO-ORCH-monitoring-parallel)へ分割済。本 US の N は単一 worktree 内での claude 子プロセスの並列起動上限を指す。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## この US 固有の 棄却した案

### R-01 — 起動を keep-alive(常駐ループが状態を握る)で実装する
- **棄却理由**: 設計 §2「プロセスは使い捨て / DB が唯一の真実 / 起動毎に突合復元」。常駐ループに状態を持たせると再起動で消え、自己復帰が成立しない。
