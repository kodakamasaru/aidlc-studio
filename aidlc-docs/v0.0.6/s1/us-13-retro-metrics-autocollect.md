## 3 視点

### なぜするか (Why)
S11(振り返り)の入力(検出ゲート別バグ数 / Step 別所要 / Q&A 件数 等)を今は人手で拾っている。run/HumanTask の store には既にこれらの素データがあるのに集計レポートが無い。自動集計すれば S11 の品質メトリクスが客観データで埋まり、振り返りが感想でなく事実ベースになる。自走 core(本サイクル)で run/HumanTask の状態遷移が増えるほど、この自動集計の価値が上がる。

### UX へのインパクト
ユーザー(振り返りをする人)は、サイクル終盤に「どのゲートで何件バグが捕まったか / どの Step が長かったか / Q&A が何件あったか」を自動レポートで受け取れる。手集計の手間が消え、改善判断が早くなる。

### 受け入れ条件 (AC)
- run / HumanTask store から **検出ゲート別バグ数・Step 別所要・Q&A 件数** を自動集計するレポートを生成する。
- レポートが **S11 の入力**として供給される(S11 がこれを読んで品質メトリクスを埋める)。
- 集計は store の事実から導出(人手の二次入力に依存しない)。
