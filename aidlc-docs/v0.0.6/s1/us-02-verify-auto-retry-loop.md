## 3 視点

### なぜするか (Why)
「ちゃんとできたら次へ、ダメなら作り直す」を人間抜きで成立させるには、システムが「ちゃんと」を機械判定し、NG なら人間に出さず自動で作り直さねばならない。claude に自分の成果を自己採点させると偽合格を構造的に生む。v0.0.5 で証拠ゲート/評価者/mock 突合は「人間に出す前のブロック」として存在するが、自動 retry のトリガには配線されていない(v0.0.5 US-01 D-03 が残した接点)。

### UX へのインパクト
ユーザーには「技術的に不完全な成果物」が一切上がってこない。検証 NG は裏で作り直されるだけで、Inbox に現れるのは本物の判断依頼だけになる。責務契約②「止まってよいのは human-gate だけ」が構造で守られる。

### 受け入れ条件 (AC)
- 各技術 step(S5/S6/S7/S8/S9 等)で **生成 → 独立検証 → 通れば DB を done に前進 / 落ちれば自動 retry(generate からやり直す)** のループが回る。
- 独立検証は**生成 run とは別の run / 決定論チェック**が握る(自己採点しない / 設計 §4)。既存資産(完了条件 gap = requirements − addressed / evaluator run / mock vs 実機の vision 判定)を**自動 retry のトリガ**として配線。
- 検証 NG は**人間に出さず**作り直す(HumanTask を立てない)。
- emit はスキーマ厳格検証。壊れていたら人間でなく retry(誤った done を作らない / 設計 §7-1)。
- status=done でも**成果物不在 / 検証 NG なら done にしない**(claude の自己申告を権威にしない / 設計 §7-2)。
- 検証で発見した「実シナリオ上の本物のバグ」(= 人間の判断や受け入れに関わるもの)は retry で握り潰さず、別途のレビュー/受け入れ経路に上げる(技術的不完全さの自動 retry とは区別する)。

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — retry 回数/backoff の制御は US-03/04 に委ね、本 US は「検証→自動作り直しの配線」に限定
- **理由**: 1 US = 1 縦スライス。本 US は「機械検証を自動 retry トリガにする」スライス。上限超過時の backoff(US-03)・上限到達時の inbox 化(US-04)は別シナリオの縦スライスとして分離し、独立テスト性を保つ。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし
