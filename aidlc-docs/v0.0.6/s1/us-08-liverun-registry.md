## 3 視点

### なぜするか (Why)
「今どのサイクルのどのタスクが起動中か」を claude は列挙してくれない(設計で実測確認済)。studio が DB に稼働台帳を持たないと、再起動後に「どのプロセスが孤児か(US-05)」「どれが無音で何分経ったか(US-06 stall)」を正確に突合できない。session_id は既に永続化済だが、pid・last-activity を含む完全な稼働台帳がまだ無い(設計 §10 / §11)。これが reconcile と stall 検知の共通基盤になる。

### UX へのインパクト
ユーザーは「走っているはずのタスク」の一覧と最終活動時刻を信頼できる形で見られる。アプリを再起動しても「何が走っていたか」が DB から復元され、幽霊run や取りこぼしが起きない。自走の状態が常に DB 上の事実として残る。

### 受け入れ条件 (AC)
- **runId ↔ pid ↔ session_id ↔ startedAt ↔ last-activity** を DB に永続する(設計 §10)。in-memory に置かない(再起動で消えない / 設計 §2)。
- last-activity は US-07 の逐次監視が供給する(走行中に更新される)。
- 起動時 reconcile(US-05)が本台帳を読んで、DB の running と実 pid を突合し孤児を判定できる。
- stall 検知(US-06)が last-activity を読んで idle 時間を算出できる。
- 台帳の更新は原子コミット + 冪等(二重更新で不整合化しない / 設計 §2)。

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — 稼働台帳を US-05(reconcile)・US-06(stall)の共通基盤として独立 US にする
- **理由**: 台帳は reconcile の孤児判定と stall の idle 算出の両方が読む共有データ(設計の橋)。各々に埋め込むと二重定義になる。「今何が起動中か」を DB 事実として持つこと自体が独立してテスト可能な縦スライス。
- **種別**: 技術判断(AI 自走で確定)

## この US 固有の 棄却した案

### R-01 — 稼働状態を in-memory のプロセス表で持つ
- **棄却理由**: 設計 §2「実行中状態を in-memory に置かない(置いたら再起動で消える)」。DB が唯一の真実。再起動復帰(US-05)が成立しなくなる。
