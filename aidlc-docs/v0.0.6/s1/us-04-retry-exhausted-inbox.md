## 3 視点

### なぜするか (Why)
無限 retry はトークンを無限消費する。一方で失敗を握り潰すと納品物がこっそり欠ける。throughput(他タスクの前進)も止めたくない。この三方を同時に満たすには「上限まで失敗した run は inbox に積み、スケジューラは止まらず後続へ進む」しかない(設計 §6 / ★確定方針)。

### UX へのインパクト
永久に直らない不良も「要対応カード」として可視化され、人間がいつでも拾える(失われない)。同時に、その 1 件のせいで他のタスクが止まることはなく、放置度が保たれる。これは routine な human-gate ではなく**例外通知**(通常フローでは発生しない最後の砦)。

### 受け入れ条件 (AC)
- retry 上限まで失敗した run は **inbox に「要対応」カード**として積まれる(人間がいつか拾える / 失われない / 設計 §12-4)。
- **スケジューラは止まらず、N の空きで後続の eligible タスクを起動**する(設計 §6 / §12-4)。
- inbox カードは「どの Cycle のどの段階が、何回試して何で落ちたか」を**事業語で**説明する(ソース内部語・パスを出さない / 責務契約①)。
- これは技術的失敗の例外通知であって、routine な human-gate(要件/画面/レビュー/改善)とは区別して表示される。

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — backoff-retriable(上限/レート)は本 US の「inbox 落ち」対象に含めない
- **理由**: US-03 で上限/レートは別カウンタの backoff 自動再開とした。本 US が拾うのは「作り直しても直らない不完全成果物」が retry 上限に達したケース。上限/レートは時間で回復するので inbox に落とさない。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし
