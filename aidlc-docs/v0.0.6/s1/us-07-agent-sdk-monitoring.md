## 3 視点

### なぜするか (Why)
今の実行は claude を CLI 子プロセスとして起動し、出力をプロセス終了後に一気読み(塊読み)している。走行中は無言で、出力が「無出力でハングしている」のか「大量にバッファ中」なのか区別できない。idle stall 検知(US-08)・last-activity による孤児精緻化(US-06 稼働台帳)は、走行中の逐次観測が無いと原理的に作れない。これを Agent SDK `query()` に載せ替え、構造化された進捗メッセージを逐次受け取れる実行基盤にする(設計 §10 / §11 / §13 の監視層を v0.0.6 へ前倒し)。

### UX へのインパクト
ユーザーから見て「今このタスクが何をしているか(走行中 / ツール実行中 / 何分無音か)」が正確に把握でき、ハングが即座に検知されて retry に回る。塊読み時代の「終わるまで何も分からない」が解消され、放置自走の観測性が上がる。

### 受け入れ条件 (AC)
- live 実行経路を **CLI spawn の塊読みから Agent SDK `query()`(または逐次 stream 読み)へ移行**し、走行中の進捗/状態を**逐次**取得する(emit を待たない / 設計 §10)。
- 無出力時間(idle)を走行中に計測できる(US-08 stall の idle timeout / last-activity の供給源になる)。
- 既存の live 実行(launch / retry / eval / resume)が全て新基盤で動く(回帰なし / 旧経路の挙動を維持)。
- 失敗分類(US-03 の上限/レート判定)・session_id 抽出・evidence 自動生成が新基盤でも従来どおり機能する。

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — 監視層(逐次監視 + 稼働台帳)を v0.0.6 に前倒しし、worktree 真の並行(N>1)だけ v0.0.7 に残す
- **理由**: ユーザー判断(2026-06-21 / 全体 Q-02)。idle stall 検知(US-08)が「本物」になるには走行中の逐次観測が必須で、塊読みのままでは壁時計 timeout しか作れず設計§5/§8 の「ハング(無出力)→ idle timeout」を満たせない。監視は自走 core に load-bearing。一方 worktree 複数(書込競合分離・マージ戦略)は重インフラで core が乗ってからでよい。
- **種別**: 事業判断(ユーザー確定)
- **上書き**: なし

### D-02 — 移行は「実行基盤の載せ替え + 逐次監視」までで、worktree 並行・組織コスト OTEL は含めない
- **理由**: 1 US = 1 縦スライス。本 US は「走行中を逐次観測できる実行基盤」のスライス。稼働台帳の永続(US-08)・stall 判定(US-08 stall）は別スライス。OTEL(組織コスト可視化)は設計§10 で任意・保留可。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし
