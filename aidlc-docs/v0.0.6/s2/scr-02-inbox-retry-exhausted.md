# SCR-02: Inbox「要対応」例外カード(retry 上限到達)

## メタ
- 親: 画面要素の一覧
- 対応 US: US-04(retry 上限到達 → inbox + 後続継続)
- ステータス: 確定

## 目的
作り直しても直らず retry 上限に達したタスクを、既存 Inbox に「要対応」例外カードとして出す。永久不良を失わず可視化しつつ、これが routine な判断依頼(Q / 視覚レビュー / 手戻り / 実機確認)とは別物だと一目で分かるようにする。

## 主要要素
- **カード種別ラベル**: `要対応(自動復旧できず)` — 既存の Q / レビュー / 手戻り / 実機確認カードと視覚的に区別(例外であることを明示)。
- **事業語の説明**: どのサイクルのどの段階が、何回試して、何で落ちたか(内部語・パスは出さない)。
- アクション: `手動で再実行` / `戻って直す`(真因が上流なら手戻り)/ `保留`(カードを未対応から外すのみ。step は failed のまま・依存は parking・スケジューラは自動で他タスク継続)+ `詳細を見る`(会話スレッドの「retry 上限到達 詳細」(SCR-05)へ navigation。SCR-05 では試行履歴と同じ 3 アクションを提示)。「後続を優先」は設計§6 で自動なのでボタン化しない(旧「スキップ」廃止 / 下 D-03)。
- **非ブロッキングの明示**: 「このカードがあっても他のタスクは止まっていません」の一言(放置度の担保を人間に伝える)。

```
|  Human Inbox                                          未対応 4 件      |

```

## この画面固有の 質疑応答ログ

### Q-01 — 「要対応」カードのアクションは何にするか(旧「スキップ」の意味が曖昧)
- **回答**(人間の回答を AI が記入):
  > スキップを押したらどうなるのか(S3 視覚レビューで指摘)→ 手動再実行 / 戻って直す / 保留 にする。
- **確定**(AI 記入):
  > 旧「スキップして後続を優先」を廃止(後続継続は§6 で自動 = 二重操作 / 必須 step を飛ばして done は契約③違反)。アクションは **手動で再実行 / 戻って直す(上流へ手戻り)/ 保留(カードを外すのみ・step は failed のまま)** + 詳細を見る(navigation)。D-03。

## この画面固有の AI が独自に決めたこと と 理由

### D-01 — 例外カードを既存 Inbox 内に置き、種別ラベルで区別する
- **理由**: 設計 §6「routine な human-gate ではなく例外通知」。別画面に隔離すると人間が見落とす。同じ Inbox に置きつつ `要対応` ラベル + 警告色で「通常フローでは出ない最後の砦」だと伝えるのが、可視化と区別の両立。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-02 — 「他のタスクは止まっていない」を明示する
- **理由**: 設計 §6 の非ブロッキング(詰まった 1 件で全体を止めない)を人間に伝え、不要な焦り/全停止の誤解を防ぐ。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-03 — アクションは 手動で再実行 / 戻って直す / 保留(+ 詳細を見る)。旧「スキップ」廃止
- **理由**: 旧「スキップして後続を優先」は semantics が破綻していた。① 後続継続は設計§6 で自動なのでボタンは二重操作 ② 必須 step を飛ばして done 扱いは `done=納品`(契約③)違反。正しい解決は 手動で再実行 / 戻って直す(真因が上流なら手戻り)/ 保留(カードを未対応から外すのみ・step は failed のまま・依存は parking・スケジューラは自動で他タスク継続)の 3 つ。詳細を見るは解決でなく navigation。S3 視覚レビューでユーザーが指摘し確定(2026-06-21)。S3 SCR-02 と同期。
- **種別**: 事業判断(ユーザー確定)
- **上書き**: 旧「スキップ」を「戻って直す / 保留」に置換。
