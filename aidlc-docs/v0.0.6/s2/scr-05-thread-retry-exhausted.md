# SCR-05: 会話スレッド(retry 上限到達 詳細)

## メタ
- 親: 画面要素の一覧
- 対応 US: US-04 (retry 上限到達 → inbox + 後続継続) / US-05 (reconcile-resume) / US-06 (stall 検知)
- 遷移元: SCR-02「詳細を見る」
- ステータス: 確定

## 目的

SCR-02 要対応カードの「詳細を見る」を押した後の詳細ビュー。自動 retry が上限に達して終了した step について、試行の履歴と各回の失敗理由を事業語で人間に見せる。人間は 3 つのアクション(手動で再実行 / 戻って直す / 保留)のいずれかで対処できる。

## 主要要素

- **ヘッダ**: 「‹ Inbox」戻りリンク / ステップ名(`ドメインモデル`)/ サイクル+ステップ番号(`v0.0.6 / S6`)/ 状態バッジ `⚠ 要対応(自動復旧できず)`(attention 赤)
- **スレッド要約行**: 何を生成しようとして何が起きたかを事業語 1 文で表示。内部パスや ID は出さない(責務契約①)。
- **試行履歴パネル**: 「試行履歴」見出し付き。3 行のリスト。各行に「試行 N」ラベル + 「自動」バッジ(人間に出さず AI が自動でやり直した) + 事業語の失敗理由。
- **最終結果ボックス**: 「3 回作り直しましたが、完了条件を満たせませんでした。」+ 「(他のタスクは止まらず進行中)」非ブロッキング注記(US-04 AC 準拠)。
- **アクションバー**: `手動で再実行`(primary) / `戻って直す`(secondary) / `保留`(ghost)。スキップなし・詳細を見るなし(すでに詳細ビュー内 / SCR-02 D-03 と整合)。

## 対応 US

| US | 対応する要素 |
|---|---|
| US-04 | 試行履歴(何回試して何で落ちたか / 事業語) / 最終結果ボックス / 非ブロッキング注記 / 3 アクション |

## この画面固有の AI が独自に決めたこと と 理由

### D-01 — SCR-05 は既存 stall ビューとは別の独立画面とする

- **理由**: 既存の「行き詰まり(stall)」ビューは単発 resume 失敗の「再試行」1 ボタン表示。SCR-05 は retry 上限到達後の終端状態で、試行履歴(3 行)+ 3 アクション + 非ブロッキング注記が必要。同画面内 state 追加は情報構造が混在するため独立画面が正しい分離。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-02 — 失敗理由は事業語のみ。スタックトレース・内部 ID は出さない

- **理由**: 責務契約①「ソース未読の IT 人材が画面と説明だけで答え/判断できるか」。内部ログは非コード人の判断材料にならない。事業語の要約(「集約境界が 1 つ未定義」等)が人間の意思決定に必要な最小単位。
- **種別**: 技術判断(責務契約① に基づく / AI 自走で確定)
- **上書き**: なし

### D-03 — 「自動」バッジを各試行行に付与する

- **理由**: 「人間が気づかないうちに 3 回作り直していた」という文脈を明示する。US-04「何回試した」の AC を満たすとともに、人間が「なぜこんなにかかったのか」を誤解しないようにする。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

## この画面固有の 棄却した案

### R-01 — retry 回数を SCR-02 カードだけで完結させ、SCR-05 を設けない

- **棄却理由**: SCR-02 カードは Inbox の一覧の中にある。3 回の失敗理由を 1 行カードに詰め込むと可読性が崩れる。「詳細を見る」で詳細ビューへ飛ばす設計は、Inbox の情報密度と詳細の両立のベストプラクティスに沿う。
