# SCR-05: 会話スレッド(retry 上限到達 詳細) — v0.0.6 S3

## メタ
- 工程: S3 (UI Design)
- 対応 S2 画面: scr-05-thread-retry-exhausted (S2)
- 対応 US: US-04 (retry 上限到達 → inbox + 後続継続) / US-05 (reconcile-resume) / US-06 (stall 検知)
- ステータス: 確定
- 視覚ファイル: scr-05-thread-retry-exhausted.html
- スクショ: screenshots/scr-05-thread-retry-exhausted.default.png

## 目的

SCR-02 要対応カードの「詳細を見る」を押したときに開くビュー。自動 retry が上限に達して終了した step の試行履歴と最終失敗理由を人間が把握できるように提示し、3 つの解決アクション(手動で再実行 / 戻って直す / 保留)を提供する。

既存の「行き詰まり」スレッド(単発 stall の resume 失敗ビュー)とは**異なる終端状態**であり、複数回の自動作り直し履歴が残る点が固有。

## レイアウト(上から順)

1. **ヘッダ帯**: ‹ Inbox 戻りリンク / ステップ名(`ドメインモデル`) / サイクル + ステップ番号(`v0.0.6 / S6`) / 状態ピル `要対応(自動復旧できず)` (--color-attention / ⚠ ドット)
2. **スレッド要約行**: 何を生成しようとして何が起きたかを事業語で 1 文。
3. **試行履歴パネル**: 「試行履歴」ヘッダ付きパネル。3 行のリスト(試行 1 / 2 / 3)、各行に「自動」バッジ + 事業語の失敗理由。
4. **最終結果ボックス**: `.exhausted-box` (attention 色)。タイトル「3 回作り直しましたが、完了条件を満たせませんでした。」+ 非ブロッキング補足「他のタスクは止まらず進行中です。」
5. **アクションバー**: [手動で再実行](primary) / [戻って直す](secondary) / [保留](ghost)。スキップなし・詳細を見るなし(すでに詳細ビュー内)。

## アクセシビリティ

- **フォーカス順序**: ヘッダ戻りリンク → アクションバー(手動で再実行 → 戻って直す → 保留)。試行履歴リストはフォーカス対象外(情報表示のみ)。
- **試行履歴リスト**: `<ol>` + `aria-label="自動試行の履歴"` でスクリーンリーダーが「1 番目、試行 1 自動 集約境界が…」と読み上げる。
- **色を唯一の信号にしない**: 状態ピルは色(attention red)+ テキスト「要対応(自動復旧できず)」の二重エンコード。最終結果ボックスは色 + タイトルテキストで区別。試行バッジは色 + 「自動」テキストで区別。
- **role / aria**: `.exhausted-box` に `role="status" aria-live="polite"` を付与し、状態変化を読み上げ可能にする。アクションバーに `role="group" aria-label="対処アクション"` を付与。

## モーション

- 状態切替・ホバー: `transition: background 150ms ease-out, border-color 150ms ease-out`(既存 .btn / .card パターンを踏襲)。
- 試行履歴パネル: アニメーションなし(情報が静止している方が読み取りやすい / prefers-reduced-motion 配慮も不要)。
- ヘッダ状態ピル: 既存 `.pill` と同一(transition なし)。

## ジェスチャー

- タップ / クリックのみ。スワイプ / 長押しは未定義(v0 スコープ外)。

## この画面固有の AI が独自に決めたこと と 理由

### D-01 — SCR-05 は既存 stall ビューとは別の終端状態として独立した section で定義する

- **理由**: 既存の「行き詰まり(stall)」ビュー(v0.0.4 の data-state="stall")は単発の resume 失敗を表し「再試行」ボタン 1 つ。SCR-05 は自動 retry が上限に達した後の終端状態で、試行履歴リストと 3 つの解決アクションが必要。同一 section に state="stall" で流用すると情報構造の意味が崩れるため独立ファイルに分離。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-02 — 試行履歴の各行に「自動」バッジを置く

- **理由**: 「これは人間が知らないうちに AI が自動でやり直していた」ことを人間が読み取れるようにするため。US-04 の AC「何回試して何で落ちたか」と責務契約① の「内部語を出さない」を両立する。「自動」は事業語として自己完結する。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-03 — 最終結果ボックスに「他のタスクは止まらず進行中」を非ブロッキング注記として置く

- **理由**: US-04 AC「スケジューラは止まらず後続継続」を人間が見て理解できるよう文字化。SCR-02 の D-02 と同じ理由。スレッド詳細まで開いた人間は焦りやすいため、この文脈で再確認できることが重要。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-04 — アクションは 手動で再実行 / 戻って直す / 保留 の 3 つ。スキップなし・詳細を見るなし

- **理由**: SCR-02 D-03(ユーザー確定)と整合。「詳細を見る」は SCR-02 から SCR-05 への navigation リンクであり、SCR-05 内に置くと循環して意味をなさない。スキップは契約③(done=納品)違反。この 3 つで解決の全経路をカバーする。
- **種別**: 事業判断(SCR-02 D-03 に準拠 / ユーザー確定済み)
- **上書き**: なし

### D-05 — 責務契約① への準拠(内部語・パス・ID を画面に出さない)

- **理由**: 試行の失敗理由は事業語のみ(例:「集約境界が 1 つ未定義のまま生成された」)。session_id / runId / pid / スタックトレース / ファイルパスは一切出さない。ステップ識別子は「v0.0.6 / S6 ドメインモデル」という事業語レベルの表記に限定する。
- **種別**: 技術判断(責務契約① に基づく / AI 自走で確定)
- **上書き**: なし

## 棄却した案

### R-01 — 既存 stall ビューに data-state を追加する形で SCR-05 を実現する

- **棄却理由**: stall ビューはシンプルな「再試行」1 ボタンで設計されており、試行履歴リスト + 3 アクション + 非ブロッキング注記を追加すると情報設計が混在する。独立 section・独立ファイルの方がS7/S8 での参照が明確。

### R-02 — 失敗理由に「詳細ログを展開して見る」リンクを付ける

- **棄却理由**: 責務契約①「ファイルパス / 内部 ID / 関数・型名は人間向け出力に出すの禁止」。詳細ログは内部語を含み非コード人が読んで判断材料にならない。事業語の要約だけで十分。
