## 3 視点

### なぜするか (Why)
Human Inbox の魂 = 「AI→人間の質問はカード化」。だが live では実 Claude が(回答を要する)質問を投げていても、headless `claude -p` が mid-run 停止せず、live adapter は成功時に常に `ResultEmitted → visual_review` を出すため、出力に含まれる質問が `question` 経路に乗らない。質問がレビュー待ちに紛れ、魂が live で未成立。

### UX へのインパクト
AI が方針を聞きたいとき、それが「レビュー(◎)」ではなく「質問(?)」カードとして Inbox に立つ。人間は「これは答えるべき質問だ」と一目で分かり、回答 UI(選択肢/自由入力)で応じられる。

### 受け入れ条件 (AC)
- live run の出力に「人間への質問」が含まれるとき、それが `visual_review` でなく `question` カードとして Inbox に出る。
- 質問でない通常の完了出力は従来どおり `visual_review`(レビュー待ち)として出る(誤分類しない)。
- 1 run が複数の質問を含むとき、複数の `question` カード(または順次提示)として扱える。
- scripted アダプタでも同じ経路を再現でき、決定論テストで検証できる(live/scripted の2アダプタ整合)。
- **質問とコンテキストは決まったフォーマット(テンプレート)で出る**(index D-04)。自由文に紛れさせず、機械が確実に質問を取り出せ・人間も読みやすい形にする。具体テンプレ設計は S2/S4。
