## 3 視点

### なぜするか (Why)
サイクルの進捗バー([PhasePipeline.tsx](../../../web/src/features/cycle-detail/PhasePipeline.tsx))は現状 S1〜S12 を**横一列の固定 12 ノード**で描く。だが実際のステップ数は可変(任意ステップ S4 の有無、将来のステップ増減)で、12 ノードを横並びにすると幅が足りず窮屈/はみ出しになり「今どこにいるか」が読みにくい。人間は IDE を開かずボードだけ見て意思決定する立場なので、進捗の現在地が**一目で・崩れず**読めることが必須。

### UX へのインパクト
人間はサイクル画面で、12 ステップが 5 つの PhaseGroup(Discovery / Design / Build / Validation / Improvement)に束ねられた進捗を見る。ステップ数が増減しても各 PhaseGroup の帯が伸縮して吸収し、横溢れしない。現在の PhaseGroup とステップが強調され、完了/進行中/未着手/手戻りが色+記号で一目で分かる。

### 受け入れ条件 (AC)
- ステップ数が可変(あるサイクルに無いステップがある / 本数の増減)でも進捗表示が破綻せず、現在地が一目で読める。
- **ステップは固定の番号(S1〜S12)ではなく「名前」で表示する**(要件ヒアリング / UIデザイン / …)。番号で見せると、欠けたとき「欠番」か「振り直し」のどちらかで必ず破綻する(振り直すと methodology の工程名と食い違う)。名前なら欠番も振り直しも省略ラベルも発生しない。→ Q-03 / D-02。
- **実在するステップだけを描く**(`cycle.phases` 駆動)。「このステップは省略された」という記録はデータに無いので、不在ステップを参照する表示(例「○○ 省略」)は出さない。
- ステップを工程グループ(要件 / 設計 / 実装 / 検証 / 改善)に束ねて表示し、横一列の窮屈/はみ出しを防ぐ。本数が増減しても帯が吸収する。
- 各ステップの状態を **色 + 記号**で二重符号化:完了(✓)/ 進行中(●)/ 行き詰まり(!)/ 未着手(○)/ 手戻り再入(↩)。色のみに依存しない。
- 現在の工程グループ・現在のステップが視覚的に強調される。
- 完了済みと未着手の工程グループが一目で区別できる(帯の塗り/コネクタ)。
