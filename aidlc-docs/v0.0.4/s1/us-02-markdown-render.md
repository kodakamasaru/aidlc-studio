## 3 視点

### なぜするか (Why)
live を本物にすると、成果物本文は md(見出し・箇条書き・コード・表)で返る。今のレビュー画面(`ReviewBlocks.tsx` の summary block)は body を素のプレーンテキストで `<p>` に流すため、構造が崩れて読めない。harness 原則③「人間がコードを見ずに承認できる成果物」は、成果物が読める描画が前提。

### UX へのインパクト
人間が IDE で md を開かなくても、サイトのレビュー画面で成果物を正しい構造(見出し/箇条書き/コードブロック)で読める。承認/差し戻しの判断が web 上で完結する。

### 受け入れ条件 (AC)
- summary block の body が Markdown として描画される(見出し・箇条書き・番号付き・コードブロック・表・リンクが崩れない)。
- 描画は安全(生 HTML 注入を許さない / XSS 対策済のサニタイズ)。
- 既存の典型ブロック(ac-map / screenshot / risk / test / coverage / diff)の描画を壊さない。
- 描画は決定論的で、オフライン(外部 CDN 非依存)で動く(既存方針 §E: Tailwind CDN 不採用と整合)。
