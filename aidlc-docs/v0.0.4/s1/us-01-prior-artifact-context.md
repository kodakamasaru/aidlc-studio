## 3 視点

### なぜするか (Why)
live の AI が「前工程を踏まえた本物の成果物」を出すには、brief だけでなく前段 step の成果物(S1→S2→…の連鎖 = `aidlc-docs/{v}/sN/*`)を実際に渡す必要がある。今は受け口(`PromptComposer.contextPaths`)はあるが既定 brief のみで、サイクル文脈に応じて前段成果物 path を解決して渡す呼び出し側(live/engine)が無い。これが揃わないと後続 US の「会話」も中身の薄い成果物の上で回ることになる。

### UX へのインパクト
人間が web で見るレビュー成果物が「前の工程の結論を引き継いだ、つながった内容」になる。各 step が孤立した出力でなく、サイクルの文脈を反映するため、レビューで「前回決めたことが無視されている」差し戻しが減る。

### 受け入れ条件 (AC)
- ある step を live 起動したとき、その step の prompt に **brief + 当該サイクルの前段 step 成果物**が含まれる(contextPaths が前段成果物 path に解決される)。
- 前段成果物が存在しない/欠落しているときは、既存どおり可視マーカー(「※ 前段文脈が見つかりません」)で明示され、黙って欠落しない。
- どの前段成果物を入れるかの解決ロジックが app/engine 側にあり、step 個別のハードコードでない。
- 既存の brief 注入・3-source 合成テストを壊さない(後方互換)。
