**最後の砦である人間が、自動検証 Step の約 7 倍のバグを発見した。** これが「バグ多すぎ」の定量的正体。バグの絶対数が多いこと以上に深刻なのは、**バグの検出が品質ハーネスの自動ゲートをすべてすり抜け、人間の手作業レビューに集中した**という分布そのもの。

### 根本原因(なぜ自動ゲートが全て見逃したか)

決定論テストと E2E は **composition-root に mock を注入して**動く。一方バグが棲んでいたのは、テストが一度も通っていなかった層に集中している:

1. **live/headless の縦経路**(launch→question→answer→resume→review→approve)を実 AI で一度も通していなかった(P10)。配線漏れ・session 永続・resume cwd・version 注入(F-9/F-11/F-15/F-17)が全部ここ。
2. **binding ルールの「配送」**: 契約・運用モデルを **リンク参照**で済ませ、headless プロンプトに本文注入していなかった(P11/P12/P14/P16/F-6/F-7/F-10/F-11)。「リンクで指す」≠「AI に届く」。
3. **人間向け文字列のパス露出**(F-10/F-13/F-19/F-23 / P17/P19)。決定論テストは fixture が事業語なので構造的に検出不能。
4. **失敗経路**(malformed エンベロープ→生 dump→retry されず)を成功経路と同じ厳密さで設計していなかった(F-13/F-20/F-21/F-22 / P20/P32/P34)。
5. **AI の自己採点・自己ゲート**(mock 突合を AI が自己申告 / status:done で人間ゲートを AI がスキップ)(F-15 / P15/P27)。
6. **E2E が片分岐・直接遷移**で、実入口フロー・分岐網羅・画面間整合を歩いていなかった(F-14/F-16 / P22/P28)。

**一言で言えば**: 品質ハーネスの自動ゲートは「決定論的に書ける層」だけを守り、**実 AI の縦経路・headless へのルール配送・人間向け提示**という最もバグの出る3層をまるごと盲点にしていた。だから 22 件全部が、それらを唯一実際に通す S10(人間の手操作)まで生き残った。[[harness-quality-vision]] の原則(視覚+シナリオ確認 / 漏れ無し / コードを見ずに承認できる成果物)が、まさに自動化されていなかった箇所で破れていた。

> **【真因の訂正 — P36 参照】** 上記「自動ゲートが live 縦経路を盲点にしていた」は**症状であって真因ではない**(ユーザー指摘 2026-06-20 で是正)。真因は **人間が初日から繰り返し「live で確認せよ」と明示指示し、責務契約(P1-P9)も焼き込み済みだったのに、AI が毎回 live を S10 まで deferral した**こと = 盲点でなく**不遵守(compliance failure)**。テキストのルール追加では行動が変わらないことが本サイクルで実証された(契約を焼いた直後に同じ逆転が再発)。よって対策は「契約/S9 にもう一文足す(soft)」ではなく「**live 証拠が無ければ step を done にできない機械ハードゲート(hard)+ live を安くして deferral の理由を消す**」。詳細・Try39/40 は下部 **P36**。

### もう一つの所見 — 同一クラスのバグが兄弟経路で再発した(モグラ叩き)
- F-13(aidlc-result の生 JSON dump)を直したが reconstruction 経路を一般化せず → **F-13 が S2 で再発**(P21/T20)。

- **T-B**: **binding ルールは composer 本文注入で配送**(T10/T12/T16)。新しい `kit/rules/*.md` を足したら headless(注入)と IDE(本文/リンク)の両経路で実際に届くか probe する。
- **T-C**: **欠陥クラスを直したら兄弟経路を同時棚卸し + 禁止と正しいやり方をワンセット**(T20/T37)。
- **T-D**: **AI に自己採点・自己ゲートさせない**(T13/T25)。oracle はビルダーと別 run の evaluator が握り、停止/前進はプラットフォーム設定が決める。
- **T-E**: **AI 出力パーサは契約が誘発する自然な変種を受理する robust parser**(T33/T34)+ **失敗は人間に投げる前に仕組み内で1回直させる**(T36)。3層(robust受理→自己修復→人間 retry 無制限)を既定に([[retry-validation-architecture]])。
- **T-F**: **メトリクス自動収集**: バグ検出ゲート分布・Step 別所要・Q&A 件数を run/HumanTask store から自動集計し S11 入力にする(Problem #5 対策)。

## AI が独自に決めたこと と 理由

### D-01 — 「バグ多すぎ」を S11 の見出し所見として最上段に置き、検出ゲート別バグ件数で定量化した
- **理由**: S11 役割定義(矮小化しない / 都合よく解釈しない)+ ユーザー指摘の中心。バグの絶対数でなく「検出が人間の最終ゲートに集中した分布」こそが構造的問題であり、それを先頭に据えるのが正直な振り返り。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**(レビューで人間が覆したときのみ AI が記録): なし

### D-02 — 個票(P1〜P35 / T1〜T38)は保持し、S11 本実行では「バグ多すぎ」に直結するクラスへ集約する形で Keep/Problem/Try/改善提案を編成した
- **理由**: 既に発生時メモとして根本原因+出典付きで蓄積済の個票を重複再記述せず、ユーザー指摘に答える「分布と構造」の分析に集約価値を置いた。完了条件(メトリクス/タイムライン/3+改善提案/全バックトラック根本原因)は別途充足。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-03 — 改善提案 #1(S9 への live 縦経路 e2e ゲート必須化)を最優先と判定した
- **理由**: 22/23 のバグが「決定論緑のまま live でのみ露呈」かつ「S10 まで生存」。検出を S9 へ前倒しする gate が、バグ分布を是正する唯一の構造的レバー。既存 carried([TEST-ENV-seeded-cycle-states] / [AUTO-ORCH-durable-self-healing])と整合し追加コストも小さい。
- **種別**: 技術判断(AI 自走で確定)。ただし「各サイクルに live e2e の実行コストを課す」運用変更は事業判断の要素を含むため、S12 で人間に確認する候補として明示。
- **上書き**: なし
