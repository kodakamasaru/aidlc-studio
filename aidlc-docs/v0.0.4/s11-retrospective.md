### Problem 追補 — 質問パーサが過度に厳格で valid-intent を壊し 3 連 stall(S10 実機 F-20 / 2026-06-19)

| # | 問題 | 根本原因 | 出典 |
|---|------|---------|------|
| P32 | S3 のヒアリング回答後、再開で **3 回リトライしても全て「回答が壊れている」**になり進めない。 | AI が ```aidlc-question``` フェンスに **裸の単一質問オブジェクト** `{id,prompt,options,…}` を出した(契約の「aidlc-question schema: id/prompt/…」をそのまま 1 オブジェクトで書いた自然な形)。`parseQuestionBlock` は **`{questions:[…]}` ラッパー必須**でそれ以外を schema エラー → malformed→stall。**リトライしても AI は同じ自然な形を出す**ため毎回同じ所で stall。over-strict な parse が valid-intent を壊す [[T20]]/F-13 と同クラス。実 sandbox セッションの最終出力 2 本を旧パーサに通して schema エラー、修正後パーサで正しく 1 問にパースされることを確定。 | S10 実機(S3 ヒアリング再開)|

- 対処: `parseQuestionBlock` を robust 化 — live モデルが自然に出す 3 形すべてを受理(① `{questions:[…]}` ② 裸配列 `[{…}]` ③ 裸単一オブジェクト `{id,prompt,…}`)。後方互換。回帰テスト2件 + 実データ再パースで実証。652 det green。
- T33: **AI 出力をパースする境界は「契約が示す自然な変種」を受理する robust parser にする(prose は厳格、構造は寛容)**。契約に「id/prompt/… のスキーマ」と書けば AI はそれを**裸で**書く変種を必ず出す。ラッパー有無・単数/配列・キー別名などの自然な揺れを schema エラーにすると、**リトライしても同じ揺れが再生産されて永久に stall** する(retry は出力を変えない)。許容できる形は normalize して受理し、本当に壊れているものだけ stall させる([[T20]] 「壊れていないものを壊れ扱いにしない」/ [[bug-fix-discipline]] 実データで真因確定の実践)。
- T34(検出ギャップ): この stall は決定論テストが「ラッパー形」しか食わせていなかったため未検出。**AI 出力パーサのテストは「契約が誘発する自然な変種」(裸オブジェクト/配列/キー揺れ)を必ず含める**(happy な正規形だけでなく)。

### Problem 追補 — retry の上限/不在が「人間 dead-end」と「同じ誤りの再生産」を生む(S10 実機 F-21/F-22 / 2026-06-19)

| # | 問題 | 根本原因 | 出典 |
|---|------|---------|------|
| P33 | 壊れた出力で stall した run を**人間が再試行しても 3 回で打ち止め**(以降不可)。 | `domainRetryRun` の `maxAttempt(既定3)` cap が **手動 retry を縛っていた**。INV-6「自動 retry なし(手動)」のもと全 retry が手動なので、cap が実質「人間が retry できる回数の上限」になっていた。cap の本来の目的は AUTO retry の暴走止めで、人間の明示操作を dead-end させるのは誤り。 | S10 実機(ユーザー指摘)|
| P34 | malformed 出力で即 stall → retry しても**同じ prompt をゼロから再実行**するため同じ malformed を再生産(F-20 の 3 連 stall の構造)。 | 「壊れた出力」への対処が **consumer(パーサ)側だけ**で、producer(出す側)に検証フィードバックが無かった。retry= 入力不変の再試行なので出力も不変。「仕組みの中で直す」(ユーザー方針)になっていなかった。 | S10 実機(F-20 の真の構造)|

- 対処 P33: `RetryRunCmd.manual?:boolean` を追加し manual は cap 免除。人間の再試行入口を manual 化(人間が最終 governor)。自動の自己修復は別予算で縛る。
- 対処 P34: malformed 検出時、即 stall せず**同一セッションへ検証エラーを差し戻して AI に直させる**(in-context 自己修復 / `MAX_REPAIR_ATTEMPTS=2`)。尽きたら retriable stall → 人間 retry(P33 で無制限)。3層化: `robust 受理(T33/F-20) → 自己修復(F-22) → 人間 retry(F-21, 無制限)`。
- T35: **「自動の上限」と「人間の操作」を同じ cap で縛らない**。暴走止めの上限は AUTO ループにだけ効かせ、人間の明示操作(retry/承認)は常に通す(人間が最終 governor)。「自動なら上限、手動は無制限」を既定にする([[ai-responsibility-contract]] ② human-gate / [[harness-quality-vision]] 人間判断なしに止めない)。
- T36: **AI 出力の不正は「stall して人間に投げる」前に「仕組みの中で1回直させる」を挟む**。retry が入力不変なら出力も不変なので、ただの再試行は無意味。**検証器(正本)の指摘を producer に差し戻す自己修復**を、人間に投げる前のステップにする(壊れたエンベロープ = 人間介在なしで作り直すべき典型 / F-13 の `stalled→retry` を自走化した版 / [[bug-fix-discipline]])。v0.0.5 [[autonomous-self-healing-orchestration]] の具体実装の一つ。
