## Keep / Problem / Try

### Keep(次サイクルも続けること)
1. **honest carry / silent-descope しない規律**: S9 で敵対レビュー(pr-test-analyzer)が O-04(3rd source の黙った欠落)を捕捉し、ledger に honest 化した。原則#6 が機能している。
2. **正本一元化の前提固定**: scope.md で「① が硬い前提、境界が揺れたまま ② を合成すると drift 再生産」と明記したのは正しく、BT は ① 自体の崩れではなく照合工程の欠落で起きた。境界設計そのものは堅牢。
3. **ロールバックを AC 起点でやり直した**: S10 却下後、S8 戻しで AC を 1 項目ずつ消し込み(commit 0d3934c)、再判定で全承認。やり直しの規律が正しかった。
4. **内部 US に過剰 E2E を作らない判断**(S9 D-01/R-01): UI を持たない AC を決定論+加算層で網羅し、粒度ゲーミングを避けた。

### Problem(起きた問題・課題)
1. **確定前評価 AI の省略でバックトラックが発生(BT-2)**: 内部 US が決定論テストのみで commit され、AC 欠落が S10 まで漏れた。S10 が初めて捕捉=最後まで漏れた。
2. **ユーザー明示指示が突合インベントリに無い(BT-2 の③)**: US-04/05 は AC 上は合格だったのに、ユーザーの「S8 内で実 claude 貫通」指示に未達で却下。AC だけ見るレビューでは構造的に検出不能。
3. **設計ステップが確定 binding を逆引きしない(BT-1)**: S6 がラベル所有を US-02 binding と照合せず決め、U02 実装中まで矛盾が顕在化しなかった。
4. **「実 AI 貫通」の定義が曖昧**: 生成 run の貫通か評価 run まで含むかが scope で握られておらず、却下の主因になった。

### Try(次サイクルで試す改善策)
1. **ユーザー明示指示の台帳化 + レビュー突合**(→ 改善提案 #1)。
2. **設計ステップ完了ゲートに binding 逆引きチェック**(→ 改善提案 #2)。
3. **scope に「貫通」の対象 run を明示**(→ 改善提案 #3)。
4. **確定前 evaluator の機械強制(hook/CI)を検討**(→ 改善提案 #4 / ルール化済を仕組みで担保)。
