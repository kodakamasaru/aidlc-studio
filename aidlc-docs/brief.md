## 目的 (1文)

AI-DLC(S1〜S7)を、人間が IDE を触らずサイト操作だけで回せる「**人間の判断を中心に AI-DLC を実行するランタイム**」にし、人間の関与を **Q による方針確定** と **各ステップ最終出力の視覚レビュー(+実機確認)** だけに絞る。

## 対象ユーザー
- **主要ペルソナ**: AI-DLC で個人開発を回す開発者本人(シングルユーザー)。PM / デザイナー / エンジニアの全役割を 1 人で兼ねる。
- **利用文脈**: ローカルの ghq リポ群(self-management-app 等)を AI-DLC で開発。複数アプリ・複数サイクルを並行で進めたい。デスクで Inbox を捌くスタイル。
- **困りごと(現状の痛み)**:
  - AI に自走してほしい所で人間が手動でセッションを立て直している
  - AI が人間にコードレビューを求めてくる(人間は実機+視覚だけやりたい)
  - 1 サイクルが長い /「US 15 前後」を AI が目標化し粒度を歪める
  - md が見づらく状況が掴めない
  - 前サイクルで固めた確定が次サイクルに漏れる
  - 複数サイクルを並行で回せない
  - per-item の D承認(承認/上書き/保留)のやりとりが無駄に多い

## 完成ビジョン(理想像)

backlog のような **タスク管理アプリ風プラットフォーム**。やりたいこと(Task)を積み上げ、マイルストーン(Cycle)とその中の Task の状況が一目で分かる。**サイト上の操作をトリガーに AI 実行が始まり、基本的に人間は IDE を触らない**(AI が止まったときの retry もサイトから)。質問への回答もサイトから。Wiki も AI がサイト上で管理。**人間に依頼したいことは全部サイト上で AI から指示が来る**。

AI-DLC は **ウォーターフォールではない**。手戻りは正常系で、任意ステップから過去ステップへ戻れる。各ステップは専用 AI Agent が担当し、人間は判断に集中する。

## 中核概念(完成ビジョンの語彙)

> S5 のユビキタス言語の種。集約境界・状態遷移はここでは確定しない(S5 の仕事)。

| 概念 | 意味 |
|------|------|
| **Task** | 開発要求(例: 通知機能追加)。Backlog に積める。単独では AI-DLC を開始しない。Cycle に割り当てて初めて動く(AI による割り当て提案も可) |
| **Cycle** | AI-DLC を回す実行単位(≒ 1 バージョン)。複数 Task を含む。現在ステップ / 判断履歴 / 会話履歴 / 成果物 / 手戻り履歴を保持。**複数並行実行可** |
| **Decision** | **人間の判断イベント**(ステップ承認 / 却下 / 手戻り判断 / Q回答)。履歴化し「なぜそう決めたか」を後から追跡可能にする |
| **AI判断記録** | AI が独自に決めた細部(旧 D-NN)。**記録のみで個別承認はしない**(Q-03)。Decision とは別概念 |
| **Artifact** | 各ステップの成果物(US / Mock / Flow / UoW / Context Map / Domain Model / Code Summary / Test Report 等) |
| **Agent** | ステップ専用 AI(US / Mock / UoW / ContextMap / Domain / Code / Integration)。**中身は `kit/skills/aidlc-sN`**(ゼロから作らず orchestrator が load) |
| **手戻り** | 正常系。**AI起点**(不足/矛盾を検知し提案 例: S5→S1)と **Human起点**(レビュー結果から要求 例: S6→S2)の両方 |

## スコープ
### やる(完成ビジョン全体 / 段階実装。サイクル割りは S1 で決める)
- **Backlog**: やりたいこと(Task)を積み上げ、優先順位を変える入口
- **Task → Cycle 割り当て**(人間操作 + AI による割り当て提案)
- **Dashboard(最重要画面 / 4象限)**: Backlog(未割当 Task)/ Active Cycles(実行中 Cycle と現在ステップ)/ AI待ち(Agent 実行中)/ Human待ち(質問回答・レビュー・手戻り判断待ち)
- **サイト操作 → AI 起動**: Claude Agent SDK で headless 実行(IDE 不要)
- **stall 時 retry**: AI が止まったらサイトから再実行
- **Human Inbox**: 人間への依頼が全部ここに来る。カード種別 = Q(方針確定)/ 視覚レビュー(各ステップ最終出力)/ 手戻り判断 / 実機確認
- **レビュー(コードを見ない)**: 変更説明 / 受け入れ条件 / UI screenshot / 動作確認 / テスト結果 / カバレッジ / **リスク分析** / **差分サマリ**
- **Decision 履歴**: 判断を全て履歴化し追跡可能に
- **手戻り**(AI起点 / Human起点 両方)
- **Wiki(AI 管理)**: ユビキタス言語 / Decision / ledger を AI が維持、人間は読む
- **並行サイクル**: git worktree で複数 Cycle・複数アプリを同時進行
- **設定の外部化**: 対象リポパス / ポート / モデル等を環境変数化(将来のセルフホスト公開に備える)

### やらない (明示 / 恒久的な非目標)
- マルチテナント / マルチユーザー認証 / 課金(**シングルユーザー固定**。公開する場合も「1 ユーザー 1 セルフホストインスタンス」)
- クラウド前提の専用インフラ(ローカル常駐が基本。セルフホストは env 化で備えるに留める)
- AI-DLC 方法論そのものの再設計(S1〜S7 の工程定義は踏襲、invocation だけ web 化)
- 実機ビルド/配信の自動化(実機確認は人間の責務として残す)
- IDE surface の廃止(web/IDE 両刀。web-only 純化は将来の収束)
- per-item の D承認 ceremony(AI判断記録は残すが、ゲートはステップ最終出力の視覚レビューに集約)

## 設計時の留意(S1+ で詰める / spec に無かった穴)
- **サイクル間 ledger(#5)**: Cycle 内の判断/手戻り履歴とは別に、前 Cycle の確定を次 Cycle へ漏れなく渡す *cross-cycle* reconciliation を持つ(`ledger.yml`)
- **並行サイクルの repo 書き込み競合**: 同一リポを複数 Cycle が触るときの worktree 分離 + マージ戦略
- **Agent の中身 = kit/skills**: 再発明しない

## 最初のリリースで外せない1機能

**Human Inbox 縦ループ** — 完成ビジョンの中で**最初に端まで通す 1 本の縦スライス**:1 Cycle 作成 → サイトで Phase 起動 → AI が headless 実行・方針が要るとき Q カード → ステップ完了時に最終出力を視覚レビューカードで提示 → サイトで回答/承認 → AI 再開 → stall なら retry。「人間が IDE を触らず 1 フェーズ回る」を証明する。
(残りの機能を**どのサイクルに割るか**は S1 で版タグを付けて決める。これは brief では確定しない。)

## 質疑応答ログ

### Q-01 — このプロダクトは「個人専用ツール」か、将来「他者も使うプロダクト/OSS」か?
- **回答**(ユーザー記入):
  > シングルユーザーでよい。ただしセルフホストサービスとして公開する可能性はあり。
- **確定**(AI 記入):
  > シングルユーザー固定。マルチテナント/認証/課金は対象外。ただし「1 ユーザー 1 セルフホストインスタンス」での公開可能性は残すため、それを阻害する設計(絶対パス直書き・個人固有値のハードコード)を避ける。

### Q-02 — 回す対象は「自分のローカル ghq リポのみ」で固定してよいか?
- **回答**(ユーザー記入):
  > 固定でよいが、公開考えると環境変数化とか必要？
- **確定**(AI 記入):
  > 対象はローカル ghq リポで固定。ただし Q-01 のセルフホスト公開に備え、リポパス / ポート / モデル名等の設定は環境変数化(`.env` + 例示 `.env.example`)。コードに絶対パスを埋めない。リモート/CI 連携は将来。

### Q-03 — 「人間がやること」は 実機確認 + 視覚レビュー + 質問回答 + D承認 の 4 つに限定でよいか?
- **回答**(ユーザー記入):
  > D承認のやりとりがけっこう無駄に感じた、Qで方針を確実にして、各ステップの最終出力を視覚レビューして次に行くのが自然かと。
- **確定**(AI 記入):
  > 人間ゲートは ① **Q で方針を確実に固める**(着手前)② **各ステップの最終出力を視覚レビューして次へ** ③ 手戻り判断 ④ 実機確認、に集約。**per-item の D承認 ceremony は廃止**。AI判断記録は traceability のため残すが個別承認はさせない。人間の判断イベントは **Decision** として履歴化する。

## AI が独自に決めたこと と 理由

> 注: Q-03 により以下は **個別承認を求めない**(記録のみ)。気になる点は brief 全体の視覚レビューで差し戻す。

### D-01 — 主要ペルソナを「開発者本人(全役割兼任)1 名」と仮置き
- **理由**: 現状の痛みは全て個人開発の文脈。マルチユーザーは別レイヤ。Q-01 で確定。

### D-02 — 方法論(S1〜S7)は再設計せず invocation だけ web 化
- **理由**: 工程定義は self-management-app で実績あり。変えるのは「呼び出し方」。スキルは kit/ に救出済で web/IDE 両刀。

### D-03 — 人間ゲートを「Q(方針)+ ステップ最終出力の視覚レビュー」に集約(Q-03 由来)
- **理由**: per-item D承認は対話コストが高く価値が低い。
- **波及(要追跡)**: kit/ のスキル群は現状 `### D-NN` 承認/上書き/保留 ceremony を前提に書かれている。studio の人間ゲート設計を確定したら kit/ 方法論側もこの集約モデルに整合させる(S1 以降 / 引き継ぎに記載)。

### D-04 — ChatGPT すり合わせ spec を「完成ビジョンの入力」として吸収(big-bang 設計はしない)
- **理由**: spec は brief とほぼ一致し Decision/Dashboard/Agent を鋭く具体化。ただし #10「MVP 範囲」= S1 の版タグ付けの仕事、ドメイン/画面/DB/API = S2〜S5 の仕事。工程を飛ばすのは本プロダクトの思想に反するため、概念はビジョンに吸収し設計はプロセスで生む。
