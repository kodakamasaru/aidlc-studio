## 状態網羅(= S7/S8 実装契約)
| state | 意味(平易) | 内部対応 | 人の操作 |
|-------|------------|---------|---------|
| `working` | 進行中(AI が作る + 自動チェック) | generator + deterministic gate | 不要(待つ) |
| `verifying` | 検証中(AI が中身を点検) | evaluator | 不要(待つ) |
| `sendback` | 作り直し中(AI が抜けに気づき自分でやり直す) | gap(申請なし)→ 自動差し戻し | 不要 |
| `descope-requested` | あなたの判断待ち(AI が理由つきで見送りを相談) | AI が descope 申請 | 受信箱で判断 |

## 進行モデル(フィードバック反映)
- パイプラインのノードは **ステップ名(要件 / 画面 / モデル / 分割 / 実装 / 確認)** で表示。GEN/EVAL バッジは廃止。
- 状態は色 + 状態語で表す: 進行中=青リング、検証中=紫リング。
- **抜け(やり残し)の扱い**:
  - AI の点検で「やりたかったことの抜け」が見つかり、**AI が理由を申請していない**場合 → そのまま先に進めず **AI が自分で作り直す**(`sendback`)。人間には届かない。
  - AI が「この項目は今回見送りたい」と **理由つきで申請**した場合のみ → `descope-requested` になり、受信箱に相談が届く。
- これにより「黙って descope」も「理由なく gap を放置」も起きない(品質ハーネス: 漏れ無し)。

## レイアウト / 視覚意図
- 進行ログは平易な日本語(「扱うデータと業務ルールを整理しています」)。worktree 番号などの内部語は出さない。
- `sendback`: 落ち着いた進行色(青)で「AI が自分で直しています・操作不要」を明示。何が未反映かを平易な一文で示す。
- `descope-requested`: amber。AI の見送り理由を併記し、「あなたの承認なしには進まない」と明記。受信箱への導線。

## interaction
- ログは push(SSE)でリアルタイム更新。
- `working` → `verifying` → 合格なら「できあがりの確認」(受信箱)/ 抜け(申請なし)なら `sendback` に戻り作り直し / AI が見送り申請したら `descope-requested`。
- 何度作り直しても解決しないときだけ、人間に相談(stall)を受信箱へ。

## a11y
- 状態は色のみでなく状態語(進行中/検証中/作り直し中/判断待ち)で伝える。
- `descope-requested` のカードは `role="alert"`。ログは `aria-live="polite"`。

## motion(文字で)
- 進行中→検証中: ノードリング色を 200ms ease-out で青→紫。
- 作り直し / 判断待ちカード: 200ms cubic-bezier(0.16,1,0.3,1) で fade + slide-up。

## native 固有挙動でドメイン側に影響しそうな項目
- **抜けの2分岐をモデル化**(S5/S6): 「AI が descope 申請したか」で分岐。申請あり → 人間タスク(見送りの相談、理由必須)。申請なし → 自動差し戻し(人間タスクなし)。
- **descope 申請は理由必須**: 理由のない見送りは発生しない。理由フィールドを必須にする。
- これは US-03(completeness gate → descope Question)の **精緻化**。S1 の US-03 受け入れ条件に「理由なし gap は自動差し戻し / descope は AI の理由申請が前提」を反映する必要がある(index.md 引き継ぎ参照)。

## 3 巡目の更新(フィードバック)
- 回答・見送り判断は **このサイクル(本画面)側で実施**。受信箱は横断のお知らせ一覧に徹する。`descope-requested` の導線は「このサイクルで判断する」に変更し、判断 UI(SCR-05)はサイクル文脈で開く。

## この画面固有の AI が独自に決めたこと と 理由(フィードバック反映)

### D-01 — GEN/EVAL/GATE バッジを廃止し「進行中 / 検証中 / 作り直し中」に
- **理由**: ユーザー指摘「GEN/EVAL/GATEバッジもわかりにくい」+ 回答「進行中/検証中でOK」。役割名でなく状態語で示す方が、専門知識なしで「いま何が起きているか」がわかる。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-02 — 理由なしの抜けは AI が自動で差し戻し(人間に出さない)
- **理由**: ユーザー指摘「明示的な申請なしのgapは普通にAIレビューで差し戻し」。理由のない gap で人間を煩わせない。人間に届くのは AI が理由つきで見送りを申請したときだけ。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-03 — パイプラインはステップ名で表示(S3 等の番号でなく)
- **理由**: 内部の番号より「要件 / 画面 / モデル …」の方が何のステップか直感的。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

- **棄却理由**: 理由のない gap まで人間に投げると受信箱が溢れる。理由つき申請だけを人間に出す(D-02)。
