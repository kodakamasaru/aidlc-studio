## ブロック構成(平易な見出し / 優先度順)
| 順 | 見出し(UI) | 中身 | 言い換えた内部概念 |
|----|------------|------|------------------|
| 1 | やりたかったことの 対応状況 | 要件を平易な一文で並べ、反映済みかを示す | completeness |
| 2 | 直したことの説明 | 起きていたこと / なぜ / どう直した / 再発防止(すべて振る舞い) | bugfix dossier |
| 3 | 実際に動いた証拠 | 自動撮影した画面(2 列) | screenshot |
| 4 | 変わったところ | 動きが変わった**機能名**(ファイル名でない)+ 重要度(高/中/低) | impact |
| 5 | 操作の動画 | 準備中(次バージョン) | video |

## レイアウト / 視覚意図(フィードバック反映)
- **要件は平易な一文**で示す(「作る順番を、依存関係に沿って並べる」)。内部の要件コード(US-xx, AC-xx)は出さない。
- **直したことの説明**: 「どんな不具合だったか(起きていたこと / なぜ)」「誰に・どこに影響したか」「どう直したか」「同じことが起きないように」。関数名・ファイル名は一切出さず、ユーザーから見た現象で書く。
- **変わったところ**: 影響範囲は **機能名**(「ステップ構成の保存」「進行状況の表示」)で示す。`.ts` ファイル名は出さない。重要度は 高/中/低(英語の HIGH/MEDIUM/LOW を使わない)。
- **見送りの提案(descope)ブロックは SCR-04 から削除**。見送りは AI が理由つきで申請する別フロー(受信箱 → SCR-05)に一本化したため、できあがり確認画面には出さない。
- 全ブロック縦並び(隠さない)。topbar に「これで進める / やり直してもらう」を常駐。

## interaction
- 縦スクロールで全ブロックを俯瞰。
- 「これで進める」→ 次ステップへ。「やり直してもらう」→ 確認の上 AI に差し戻し。
- 動作証拠はクリックで拡大(将来)。

## a11y
- 各ブロックは `<section aria-labelledby>`。テーブルは `<table>` セマンティクス。
- 重要度は色 + テキスト(高/中/低)。画面写真には alt(何の画面か + 撮影時刻)。

## motion(文字で)
- ブロックは上から 60ms ずつ stagger で 200ms cubic-bezier(0.16,1,0.3,1) fade + slide-up。

## native 固有挙動でドメイン側に影響しそうな項目
- **レビュー内容は「平易な説明文」で持つ**(S5/S6): 原因・影響・変更箇所は、コード参照でなく **人間向けの説明テキスト + 機能名 + 重要度(高/中/低)** をデータとして持つ。AI(evaluator/generator)がこの平易な説明を生成する責務。
- dossier の構成: 起きていたこと / なぜ / どう直した / 再発防止 / (動画)。要件の対応状況は「要件文 + 反映済みフラグ」。
- 影響範囲は **機能名の配列 + 重要度**。ファイルパスは持たない(or 持っても UI に出さない)。

## この画面固有の AI が独自に決めたこと と 理由(フィードバック反映)

### D-01 — 原因・影響を「振る舞いベース」で書く(コード・ファイル名を出さない)
- **理由**: ユーザー指摘「原因とか影響範囲とかソースコードベースで言われてもわからない。内部構造を知らなくても使えるのが目的」。現象(起きていたこと)と効き先(誰に / どの機能)で説明する。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-02 — 重要度は「高/中/低」、要件は平易な一文
- **理由**: HIGH/MEDIUM/LOW や US-xx コードは内部語。日本語の重要度と平易な要件文に置換。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-03 — descope ブロックを SCR-04 から削除
- **理由**: 見送りは「AI が理由つきで申請 → 受信箱 → SCR-05 で判断」に一本化。できあがり確認画面に混ぜると役割が二重になる。
- **判断**(ユーザー記入): 承認 | 上書き | 保留
