## 状態網羅(= S7/S8 実装契約)
| state | 意味 | 主な視覚差分 |
|-------|------|-------------|
| `list` | サイクル一覧 | ステップ構成へは **控えめなテキストリンク**(大ボタンにしない)。主ボタンは「始める / 中身を開く」 |
| `settings` | 設定 > デフォルトのステップ構成(全サイクル共通テンプレ) | **AI と対話 → AI が変更案を提案 → プレビュー → 適用**。各ステップの「やること」を平易に表示 |
| `cycle-steps` | サイクルのステップ構成(確認のみ) | 進行中は変更不可の注意 + 各ステップの役割と進捗を一覧 |
| `full-spec` | AI 実行用の指示・全文(要約の「全文を見る」から) | 役割/目的/手順/観点チェック/出力形式/例/完了条件/禁止。要約より詳細 = AI のコンテキスト本文 |

## レイアウト / 視覚意図(フィードバック反映)
- **ステップが何をやるか**を最優先で表示。各ステップ行に「やること」の平易な一文を添える(例: ドメインモデル =「扱うデータと業務ルールを整理する」)。内部の契約名は出さない。
- **デフォルトのステップ構成は「設定」の中**(全サイクルのひな形)。サイクルごとの構成は、ここを土台に作られる。
- **編集は対話式**: ユーザーが要望を書く → AI が「編集後の構成」を提案(`.proposal`)→ プレビューで差分(変更/追加マーカー)を確認 → 「この内容で適用」。手入力フォームは廃止。
- **サイクル側のステップは控えめ**: サイクルカード内の「ステップ構成を見る・調整」は `.subtle-link`(小さいテキスト)。使用頻度が低い前提で、主導線(始める/開く)を邪魔しない。
- 進行中サイクルの構成は確認のみ(変更は開始前のみ)。

## interaction
- **対話編集**: textarea に要望 → AI 応答 → `.proposal` 提示 → 「適用」で保存(次サイクル以降のひな形に反映)/「やめる」で破棄。続けて相談も可能。
- **控えめリンク**: クリックで `cycle-steps`(確認)へ。未開始サイクルなら同じ対話編集を開ける。
- 設定のデフォルトを変えても、進行中・既存サイクルには遡及しない(これから作るサイクルに適用)。

## a11y
- 対話スレッドは `role="log"`。提案カードは `role="region" aria-label="AIからの変更案"`。
- 適用/やめるは明確なラベル。プレビューの変更箇所は色(緑=追加/青=変更)+ テキストタグの二重符号。
- 控えめリンクも十分なタップ領域とコントラストを確保。

## motion(文字で)
- AI 応答・提案カード出現: 200ms cubic-bezier(0.16,1,0.3,1) で fade + slide-up。
- 適用成功: 提案カードを 200ms で done 色に変えてから「いまの構成」に反映。

## native 固有挙動でドメイン側に影響しそうな項目
- **デフォルトのステップ構成(テンプレ)** は新しい永続データ。S5 で「テンプレ」と「サイクルごとの構成」を分けてモデル化(サイクル作成時にテンプレを複製)。
- **ステップの中身**(やること / 完了の条件 / 人の確認 / 並行可否)は AI 提案を人間が承認して確定する。AI 提案 → プレビュー → 適用、の確定フローを S5/S6 の整合性に含める。
- 進行中サイクルの構成変更禁止は UI だけでなく API でも拒否(状態で判定)。

## 3 巡目の更新(フィードバック)
- 画面に出すステップ指示は **人間向けの要約**。実際に AI が使う指示は、この要約を基に **観点・チェック項目・例まで含めた全文へ展開**したもの(= 別レイヤー)。UI は要約 + 「AI 実行用の全文を見る」導線。「この要約そのものを AI のコンテキストにする」わけではない(ユーザー確認: 要約として見せるなら今の粒度でちょうどよい)。
- 要約の構成: やること / 参照する入力 / 出す成果物 / 完了の条件 / 人の確認 / 並行可否。対話編集のプレビューも要約レベルの差分で見せる。
- 言語水準: 一般エンジニア用語(テスト / 型 / 受け入れ条件 等)は使用可。この PJ のソース固有語(ファイル名・内部シンボル・集約名)は禁止([[target-user-persona]])。
- 対話編集のプレビューも詳細粒度の差分で見せる(完了の条件への項目追加など)。

## この画面固有の AI が独自に決めたこと と 理由(フィードバック反映)

### D-01 — ステップ定義の編集は「対話 → AI 提案 → プレビュー → 承認」にする
- **理由**: ユーザー指摘「AIと対話 → AIが編集後の定義を提案 → プレビュー → 承認して適用 がいい」。手入力フォームは内部概念(契約4種等)を人間に押し付ける。対話式なら「やりたいこと」を言葉で伝え、AI が定義に翻訳し、人間は結果を見て承認するだけで済む。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-02 — デフォルトのステップ構成は「設定」に置く
- **理由**: ユーザー指摘「今後のデフォルトとなるステップ定義を編集する導線がない」+ 回答「設定画面の中」。全サイクル共通のひな形は、特定サイクルから独立した常設の場所(設定)が自然。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-03 — サイクル側のステップ構成は控えめなリンクに格下げ
- **理由**: ユーザー指摘「サイクル側のステップ定義編集/確認は使用頻度低い…でかでかとしたボタンじゃない方がよい」。主導線(始める/開く)を優先し、ステップ構成は小さいテキストリンクにした。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-04 — 各ステップに「やること」の平易な一文を必ず添える
- **理由**: ユーザー指摘「一番肝心の、そのステップがどんなことをやるかがわからない」。ステップ名だけでなく役割の説明を併記し、専門知識なしで構成を理解できるようにした。
- **判断**(ユーザー記入): 承認 | 上書き | 保留
