## 想定ユーザー像(言語水準の基準)
**一般的な IT 用語はある程度わかるエンジニア。ただし、このプロダクトのソースコードは一切知らない。**
- 使ってよい: 一般的なエンジニア/IT 用語(テスト、型、受け入れ条件、依存関係、API、成果物 など)。
- 禁止(UI・人間向け文言): この PJ のソース固有語(ファイル名・内部シンボル・集約名・画面 ID・要件コード・内製略語)。
- AI 起動の土台になるステップ定義は、AI が実行できる **詳細な粒度**で書く(粗すぎると遂行できない)。

## 全体 AI が独自に決めたこと と 理由

### D-01 — v0.0.1 のトークン / styles.css を継承(Tailwind CDN 不使用)
- **理由**: capture 環境の CDN 不達 + 既存 web との一貫性。S7/S8 は html/css を読まないため影響なし。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-02 — UI 文言から内部語・コード・画面 ID をすべて排除(2 巡目)
- **理由**: フィードバック #1/#8/#10。内部構造を知らない人が読んで判断できることが目的。役割名でなく状態語、ファイル名でなく機能名、要件コードでなく平易な一文。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-03 — 見送り(descope)は「AI が理由つきで申請したときだけ人間に出す」
- **理由**: フィードバック #6。理由なしの抜けは AI レビューで自動差し戻し(人間に出さない)。黙った descope も理由なし放置も防ぐ。SCR-02/03/05 で一貫。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-04 — ステップ定義は「設定のデフォルト」+「対話編集」に再設計
- **理由**: フィードバック #3/#4/#5。デフォルトは設定に常設、編集は対話 → AI 提案 → プレビュー → 承認。サイクル側は控えめリンク。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-05 — 言語水準は「一般 IT 用語 OK / PJ ソース固有語 NG」に再調整(3 巡目)
- **理由**: 想定ユーザー像の共有(一般 IT 用語はわかるエンジニア / この PJ のソースは知らない)。過度な幼稚化は不要。一般エンジニア用語は使い、PJ ソース固有語のみ禁止する。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-06 — ステップ定義は AI 実行可能な詳細粒度で持つ(3 巡目)
- **理由**: フィードバック #13。粗い一文では AI がステップを遂行できない。やること詳細 / 入力 / 成果物 / 完了条件 / 人の確認 / 並行可否 を持たせ、これが AI 起動の土台になる。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-07 — 判断はサイクル側、受信箱はお知らせ一覧(3 巡目)
- **理由**: フィードバック #14。受信箱は横断のお知らせ一覧(件数のみ)、クリックでサイクルへ。回答・見送り判断はサイクル文脈(SCR-02/05)で行う。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

### D-08 — やり直すステップ選択に AI の推奨を表示(3 巡目)
- **理由**: フィードバック #15。戻り先ステップを人間任せにせず、AI の推奨(★)と理由を添える。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

## 次工程 (S5 / S7 / S8 / S1) への引き継ぎ
- **S7/S8 が参照すべき対応表**: 各 `scr-NN-*.md` の「視覚 source / スクショ」。S7/S8 は `screenshots/*.png` + `scr-NN-*.md` のみ。
- **状態網羅(= 実装契約)**:
  - SCR-01 {list / settings(対話編集) / cycle-steps / full-spec(AI 実行用の全文)}
  - SCR-02 {working / verifying / sendback / descope-requested}
  - SCR-03 {list / empty}
  - SCR-04 {default}
  - SCR-05 {question / descope / confirm / resume}
- **S5(ドメインモデル)へ渡す型/制約(2 巡目で更新)**:
  - **ステップ定義のテンプレ化**: 「デフォルトのステップ構成(設定)」と「サイクルごとの構成」を分離。サイクル作成時にテンプレを複製。
  - **ステップの中身は「要約(人間向け)」と「全文(AI 実行用)」の2層**: UI に出すのは要約。AI が実際に使う指示は、要約を基に観点・チェック項目・例まで展開した全文。S5/S6 で両者をモデル化(要約 ↔ 全文の対応)。AI 提案 → 人間承認の確定フロー。
  - **やり直し先の推奨は gap の原因分析から算出**(固定でない)。AI が「抜け → 原因のあるステップ」を導き、推奨 + 理由を返す。
  - **回答・判断の場所**: 受信箱はお知らせ一覧(横断・件数)。回答/見送り判断はサイクル文脈(SCR-02/05)で行う。受信箱カード → 該当サイクルへ遷移。
  - **やり直し先の推奨**: 見送り判断の「前のステップからやり直す」は AI が推奨ステップ + 理由を返す。
  - **見送り(descope)モデル**: AI の見送りは **理由必須**。発火は AI 申請時のみ。理由なし gap は自動差し戻し(人間タスクなし)。
  - **見送りの判断結果(4種)**: つくる / 見送る(持ち越し)/ 前のステップからやり直す(戻り先ステップ id 付き)/ 自由指示。
  - **回答ペイロード**: 選択肢 id または 自由入力テキスト(1本)。質問はステップ単位で複数件。
  - **レビュー内容は平易な説明文 + 機能名 + 重要度(高/中/低)** を持つ(コード参照を UI に出さない)。AI がこの平易テキストを生成。
- **★ S2 / S1 へ戻すべき項目(画面要素・要件の不足)**:
  - **設定画面 / デフォルトのステップ構成は S2 に存在しなかった新規導線**。S2(情報構造)に「設定」surface を追記して整合させる必要がある。
  - **descope モデルの精緻化**(理由必須 / 無申請は自動差し戻し / やり直し選択肢)は US-03 の受け入れ条件に反映が必要(S1 へ)。
- **orchestration → web の push**: 進行ログ / 件数 / resume はリアルタイム(SSE)。
