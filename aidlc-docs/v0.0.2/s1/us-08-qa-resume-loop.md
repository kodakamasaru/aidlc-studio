## 3 視点

### なぜするか (Why)
v0.0.1 は Run→emit→done の単方向実行のみ検証した。v0.0.2 の gen→eval→human 承認ループでは、**AI が方針を質問(Q)して停止し、人間が回答して AI が再開(resume)** する対話フローが不可欠。これは Human Inbox の核心機能であり、品質ハーネス原則#1(視覚+シナリオ確認)の前提。

### UX へのインパクト
ユーザー(開発者)は Human Inbox に Q カードが届き、回答を書くと AI が自動再開する。**ポーリングで状態を確認し、Q が届いたら HumanTask カードとして表示**。回答後、セッションが resume されて AI が続行する。IDE を一切触らず1フェーズが完結する。

### 受け入れ条件 (AC)
- [ ] live adapter が Q 出力を検知して Run を stalled 状態にする
- [ ] stalled Run から HumanTask(Q タイプ)が自動生成される
- [ ] HumanTask に Q 内容が表示される
- [ ] 受信箱は **お知らせ一覧**(種類 + サイクル/ステップ + 件数)で、判断はしない。クリックでサイクルへ遷移
- [ ] 回答・判断は **サイクル画面側**で行う
- [ ] 1 ステップで複数質問がある場合、受信箱は件数のみ表示し、回答画面で 1 件ずつ進める
- [ ] 回答は「選択肢(推奨マーク付き)+ **その他(自由入力)**」のハイブリッドで、**入力欄は1個**(その他選択時のみ展開)
- [ ] ユーザーが回答を送信すると、セッションが resume される
- [ ] resume 後の AI 出力が引き続きイベントとしてemitされる
- [ ] resume→完了/再Q停止→回答→resume のループが2周以上回ることをテストで確認
- [ ] E2E テスト(実 AI 使用)で Q→回答→resume フローが pass する
- [ ] 既存テストが全 pass

### D-02 — resume はセッション ID で再接続する(headless claude の --resume)
- **理由**: Claude CLI は `--resume <session-id>` でセッションを再開可能。Q で止まったセッションの ID を保持し、回答を入力として resume する。新しいセッションを作り直すより効率的。
