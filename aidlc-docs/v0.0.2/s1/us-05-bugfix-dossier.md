## 3 視点

### なぜするか (Why)
バグ修正のプロセスを構造化し、「なぜ起きたか / 影響範囲 / どう直すか / 再発防止」を漏れなく記録する。Bugfix dossier は evaluator が検証する成果物 Profile の一種で、原因分析(2層)・影響・修正内容・予防策・映像記録の必須 block を定義する。

### UX へのインパクト
ユーザー(開発者)はバグ修正の理由と影響を **リッチ描画(US-07)で一覧確認** できる。コードを見ずに「この修正は何に対するものか」を理解し、承認/差し戻しできる。修正の属人化が防がれ、後から「なぜこの修正をしたか」が追跡可能。

### 受け入れ条件 (AC)
- [ ] BugfixDossierProfile が定義される(cause / impact / fix / prevention / video の必須 block)
- [ ] cause は 2 層構造(直接原因 + 根本原因)を持つ
- [ ] Profile レジストリ(US-01)に bugfix タスク種別として登録される
- [ ] coerceBlocks で bugfix dossier の block 構造が前方互換で処理される
- [ ] bugfix dossier を入力とする evaluator テストが pass する
- [ ] 既存テストが全 pass
