## 3 視点

### なぜするか (Why)
全 Step の全情報を常にプロンプトに載せるとコンテキストが肥大化し、AI の出力品質が落ちる。Core 常時 + Step Payload 遅延の 2 層に分けることで、各 Run に必要最小限のコンテキストだけを組み立てられる。gen と eval でも別 payload を使うことで、generator と evaluator に適切な視点を与える。

### UX へのインパクト
ユーザー(開発者)への直接的な UX 変化は小さいが、AI の出力品質が向上する(不要なコンテキストノイズの削減)。間接的に「AIが的確な質問・成果物を出すようになる」ことでレビュー体験が改善。

### 受け入れ条件 (AC)
- [ ] Prompt 構成が 2 層に分かれる: Core(常時ロード) + Step Payload(遅延ロード)
- [ ] Core レイヤには brief / 直前 Step の成果物サマリ / ユビキタス言語を含む
- [ ] Step Payload レイヤには当該 Step のスキル(SkillDef)と入力成果物を含む
- [ ] generator 用 payload と evaluator 用 payload が異なる内容になる(gen は SkillDef + input、eval は output + verification 契約)
- [ ] 遅延ロードが lazy に機能することをテストで確認する
- [ ] 既存テストが全 pass

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — Core レイヤの内容を brief + 直前 Step サマリ + ユビキタス言語に固定
- **理由**: 全 Run で共通して必要な最小コンテキスト。brief は製品方針、直前 Step サマリは引き継ぎ、ユビキタス言語は用語統一。これより多いとノイズ、少ないと AI が迷う。
