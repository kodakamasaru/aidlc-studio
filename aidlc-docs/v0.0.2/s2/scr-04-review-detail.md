# SCR-04: レビュー詳細(更新) — v0.0.2

## メタ
- 親: [s2/index.md](./index.md)
- 対応 US: [US-07](../s1/us-07-rich-rendering.md)
- ステータス: 確定

## 目的

v0.0.1 のレビュー詳細画面(ReviewDetail)に **リッチ描画ブロック** を追加する。ユーザーはコードを見ずに、completeness table / impact table / bugfix dossier / descope card / screenshot 証拠を確認し、approve/reject できる。

> 注: 本画面に到達する時点では gap=0(全要件 addressed)。completeness table は「全件対応済みの確認証拠」として機能する。gap による fail は SCR-05(descope 判断)で完結し、本画面には来ない。ただし evaluator からの **descope 提案**(過剰実装の削減提案)はありうるため descope card は残す(Q-01 で確定)。

## 主要要素

### A. リッチ描画ブロック群 (US-07)

v0.0.1 の ReviewBlocks に以下の block タイプを追加:

1. **completeness table**: requirements ↔ addressed の照合結果テーブル
2. **impact table**: 変更影響範囲テーブル(集約/ファイル/重要度)
3. **bugfix dossier**: cause(2層)/impact/fix/prevention/video の構造化カード
4. **descope card**: 削減提案の内容と承認/却下ボタン
5. **video embed**: 動画プレーヤー(v0.0.2 は placeholder)
6. **screenshot 証拠**: verify-ui 自動生成 screenshot の動作証拠表示

### B. 描画優先度

S1 Q-03 で確定した情報優先度に従い、上から順に配置:

1. completeness table(要件カバレッジ)
2. bugfix dossier(修正理由)
3. screenshot 証拠(動作証拠)
4. impact table(影響範囲)
5. descope card(削減提案)
6. video embed(動画)

## この画面固有の 質疑応答ログ

### Q-01 — SCR-04 に gap / descope を表示することはあるのか？
- **質問の意図**: AIレビュアーが gap を検出すると hard gate で fail し、SCR-05(descope 判断)へ手戻りする。では SCR-04(承認画面)到達時には gap=0 のはずで、completeness table や descope card を出す意味があるか。
- **回答**(ユーザー記入):
  > AIレビュアーが gap/descope を検出したら、その内容とともに手戻りになる。ただし descope 提案もありうるので残す。
- **確定**(AI 記入):
  > 原則 SCR-04 到達時は gap=0(全要件 addressed)。completeness table は「全件 addressed の確認証拠」として表示する。一方、evaluator が「実装はしたが過剰/不要なので descope 提案」を出すケースがあり、その場合は SCR-04 に descope card を表示し、承認/却下できる。gap による fail は SCR-05 で完結し SCR-04 には来ない。completeness table の gap 行クリック遷移は不要(gap=0 のため)。

## この画面固有の AI が独自に決めたこと と 理由

### D-01 — 各 block を独立したセクションとして縦に並べる
- **理由**: block 種別ごとに情報密度が異なる。タブやアコーディオンで隠すと重要情報を見落とすリスクがある。縦スクロールで全 block を一覧表示し、優先度順に並べる。
- **判断**(ユーザー記入): 承認 | 上書き | 保留
- **上書き内容**(上書き時のみ):

### D-02 — bugfix dossier の cause を2層(直接/根本)で表示する
- **理由**: US-05 で cause に2層構造を定義。直接原因だけでは「なぜそれが起きたか」が追えない。根本原因まで表示することで予防策(prevention)の妥当性を判断できる。
- **判断**(ユーザー記入): 承認 | 上書き | 保留
- **上書き内容**(上書き時のみ):

### D-03 — screenshot 証拠はグリッド表示(2列)
- **理由**: screenshot が複数ある場合、縦に並べると長くなりすぎる。2列グリッドで高さを抑えつつ一覧性を確保。画面幅が狭い(320-768)場合は1列にフォールバック。
- **判断**(ユーザー記入): 承認 | 上書き | 保留

## この画面固有の 棄却した案

### R-01 — block をタブで切り替える
- **棄却理由**: タブは隠れた情報を見落とすリスクがある。レビューでは全情報を一度に俯瞰したい。縦スクロールの方が確実。

### R-02 — completeness table だけを表示して他は詳細画面にする
- **棄却理由**: 品質ハーネス原則#3(コード不要で承認)に反する。全情報を1画面で確認できることが前提。

### R-03 — video embed に実際のプレーヤーを実装する
- **棄却理由**: v0.0.2 では録画機能が未実装。placeholder で型と枠だけ作り、v0.0.3 で実体化する(US-07 D-02 で確定済)。
