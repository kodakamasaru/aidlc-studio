## 3 視点

### なぜするか (Why)
完了到達がサイクル全工程の完走を要し高コストで、各ステップの検証も「最初から実 AI を走らせる(遅い hearing 込み)」だと毎回高くつく。**だから「いろいろなステップを即確認するためのデータ」を作る**: 任意の step を、その前段成果物・当該 step の産物・証拠・状態まで予め seed しておけば、**実 AI を走らせずに即その step を検証できる**(done ゲート / 記録者≠レビュアー監査 / 視覚証拠レビュー を seed データ上で即実行)。live deferral の経済的理由も、step 検証の遅さも消える(IMP1(b))。

### UX へのインパクト
ほぼ done のサイクルや、特定 step の直前・直後状態を seed → **その step を即確認**(完了バナーの目視、証拠ゲートの発火、別 run による証拠監査、等)。毎回フルパイプラインや hearing を待たない。実 live を回す場合(端到達確認)も seed が前段を埋めるので 1 step だけ安く走らせられる。

### 受け入れ条件 (AC)(S1 再ヒアリング Q-01 確定を反映)
- **軽い実装ベースの上に、いろんな要件のサイクル群を作る**: 1 つでなく複数の example サイクル(それぞれ異なる brief / 要件)。
- **各サイクルは異なる step で止める**: あるサイクルは S2 で、別は S5 で、別は S9 で… と停止位置を散らし、**任意 step をいずれかのサイクルで即確認できる**ようカバーする。
- **全サイクルに「実際に回した時同様の適切なデータ」を持たせる**: プレースホルダ(スタブ index.md / 1x1 PNG)は不可。各 step の成果物は**その step が実際に産むものとしてもっともらしい内容**(現実的な US / 画面 / モデル / コード / テスト)、証拠は**実アプリの実 screenshot** + 実ログ相当。
- **即確認**: seed 後、当該 step の done ゲート(US-01)/ 記録者≠レビュアーの独立監査 / 視覚証拠レビューが、遅い live 実行なしに、**もっともらしいデータの上で**実行・確認できる。
- seed が studio 自身の実 aidlc-docs を汚染しない(使い捨てリポ / 隔離 / [[test-projects-use-throwaway-repo]])。
- (補助)実 live を回す経路では毎 step の証拠を自動生成する(US-04 backtrack 前から継続 / US-01 の証拠形式)。

### Q-01 — seed データの「もっともらしさ」と構成(S1 再ヒアリング / 2026-06-20)
- **回答**(人間の回答を AI が記入):
  > 実装が軽いベースを作って、そこにいろんな要件のサイクルを作る。サイクルごとに、異なるステップで止めておく。全てのサイクル用の、実際に回した時同様の適切なデータをつくる。
- **確定**(AI 記入):
  > seed は「軽い base + 多様な要件の example サイクル群(各々別 step で停止)+ 各 step が実際に産むものと同等のもっともらしいデータ(実 screenshot 含む)」を作る。プレースホルダ不可。これで任意 step をもっともらしいサイクル上で即確認できる。

### D-04 — 証拠 screenshot は実 studio ボードの実キャプチャ(限界も明記)
- **理由**: Q-01「実際に回した時同様」を満たすため 1×1 プレースホルダを廃止。`seed-suite-capture` が seed したサイクルのボードを実 studio で開いて撮った実 PNG(約 90KB / 1440×900)を証拠にする。live 経路の `captureVerifyUi` と同性質の実キャプチャ。
- **正直な限界**: seed は対象アプリ(ToDo/チャット等)の画面そのものは撮らない(それには実 live でアプリを建てる必要)。seed が証明するのは「即確認の**機構**が、もっともらしいデータ + 実 studio キャプチャ上で動く」こと。詳細は S5 unit-04 D-04。
- **種別**: 事業判断(Q-01 反映 / 2026-06-21)
- **上書き**: なし

### 実装サマリ(BT-04 / Q-01 完了)
- スイート 5 サイクル(todo-app@S2 / inventory@S4 / booking@S6 / expense@S8 / chat@S9)を `fixtures/seed-cycles/` にコミット。各々別アプリ・別 step 停止・実 run 相当のもっともらしい本文(US/画面/モデル/純粋 .ts コード/シナリオ報告)。
- `seedSuiteCore` が project ごとに隔離 seed → `seed:suite` CLI で実機確認(5 サイクル / chat 3 manifest / expense 2 manifest)。
- 実証(走らせずに即確認 / 決定論テスト 9 pass): chat@S9・expense@S8 が `FsEvidenceGate` で即 `eligible`、証拠 screenshot は実キャプチャ(>2KB ガード)、産物は実コード/実本文(`(seeded product)` 文字列なし)。

### D-03 — seed の本質は「状態」でなく「即確認のためのデータ」(S1 backtrack で是正 / BT-04)
- **理由**: 当初 S5-S8 で US-04 を「phase 状態だけ seed」+「検証は実 claude を走らせる(遅い hearing)」で実装したが、ユーザーの原意「いろいろなステップを即確認するためのデータ作り」から外れていた(S9 で実 live を回したら generator が hearing-first で別 run 監査まで到達せず、即確認になっていない)。seed は前段成果物・産物・証拠まで作り、走らせずに即検証できることが本質。S1 から再記述し S5→S9 を作り直す。
- **種別**: 事業判断(ユーザー指摘で是正 / 2026-06-20)
- **上書き**: 旧 AC「任意の**状態**に置いたサイクルから**テストを開始**できる」→「任意 step を**走らせずに即検証できるデータ**を生成する」へ一般化・強化。

## この US 固有の 棄却した案

### R-01 — seed は phase 状態だけ置き、検証は毎回実 claude を走らせる(現行実装)
