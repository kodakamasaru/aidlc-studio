## 3 視点

### なぜするか (Why)
2 サイクル連続で AI が live 確認を S10 まで deferral した真因は盲点でなく不遵守(compliance)。soft な契約追加では行動が変わらないと v0.0.4 で実証(契約を焼いた直後に同じ逆転が再発)。deferral を物理的に不能にするには、step done を「live 証拠アーティファクトの存在」で機械検証するしかない(自己申告でなく物理前提)。

### UX へのインパクト
各技術 step は、実 backend を縦に通したログ + 視覚/動作証拠が揃って初めて done になる。人間が S10 で初めて live を踏む end-loaded 構造が解体され、人間がレビューする時点で既に live 証拠が揃っている。

### 受け入れ条件 (AC)
- 各技術 step の done を「live 証拠アーティファクト(実 backend 縦経路ログ + 視覚/動作証拠)の存在」で機械検証する
- **証拠の形式は step の性質に応じて選べる: 静的 UI = screenshot / 操作・遷移 = 動画(dossier)/ backend・スクリプト = テスト結果・実行ログ。screenshot に固定しない**(brief レビュー節 / BACKLOG §A リッチ可視化 / [harness-quality-vision] と整合)
- 証拠が無ければ step を done にできない(claude の自己申告 done を拒否する)
- per-step live: S10 を待たず各 step 完了時に当該 step の live 縦断を 1 本走らせる
- 証拠アーティファクトの所在・形式(screenshot / 動画 / test-report 等)が定義され、後から人間が辿れる

## この US 固有の AI が独自に決めたこと と 理由

### D-01 — 本ゲートは standalone(証拠が無ければ done をブロック)。自動 retry エンジンとの統合は v0.0.6
- **理由**: 自走の検証→自動作り直しエンジン(旧 US-02 相当)は v0.0.6 へ分割。v0.0.5 では本ゲートが done を止めるところまで(retry は現行どおり人手)。安価な証拠生成は US-04(seeded + 安価 live)に乗せる。証拠不在=検証 NG という意味付けは保ち、v0.0.6 の自動 retry がそのトリガを拾えるよう接点を残す。
- **種別**: 技術判断(AI 自走で確定)
- **上書き**: なし

### D-02 — 証拠の形式を screenshot に固定せず step 性質で選ぶ(S2 中にユーザー指摘で補正)
- **理由**: 当初 AC を「配信 screenshot」と狭く書いたが、証拠は動画(dossier)/ テスト結果 / 実行ログ等も取り得る(brief レビュー節 / BACKLOG §A リッチ可視化 / [harness-quality-vision])。静的 UI=screenshot、操作・遷移=動画、backend=test-report のように step に応じて選ぶ。機械ゲートは「適切な形式の視覚/動作証拠が存在するか」を検証する。
- **種別**: 事業判断(ユーザー指摘で補正 / 2026-06-20 S2 中)
- **上書き**: AC の「配信 screenshot」固定 → 「視覚/動作証拠(screenshot / 動画 / test-report 等)」へ一般化
